# Глубокий сравнительный анализ двух проектов FVSC

## 1. Что представляют собой проекты

### FVSC_логический_каркас.md — теоретическая спецификация

Автономный документ (~1200 строк), описывающий **формальную лингвистическую модель** записи персональных смыслов. Это архитектурный манифест, построенный на фундаменте формальной семантики: Frege, Wittgenstein, DRT (Kamp), FrameNet (Fillmore), Neo-Davidsonian event semantics.

Ключевые принципы:
- **Контейнерная модель**: значение слова = совокупность того, что человек помещает внутрь через свои формулировки
- **Стенографический принцип**: система записывает ТОЛЬКО то, что прямо следует из слов, никакой интерпретации
- **Глагол как дуальная сущность**: одновременно связка между понятиями И самостоятельный контейнер
- **Суждение = Субъект + Глагол + Объект + свойства** (качество, модальность, интенсивность, время)

Планируемый стек: **spaCy** (dependency parsing) → **Neo4j** (граф) + **Qdrant** (векторы). Пайплайн из 7 стадий с PMI, DBSCAN-кластеризацией для граней смысла, алгоритмом Леска для омонимии.

### platform-codex-exocortexAUTOTRY — рабочая реализация

Полноценный Python-пакет (v0.1.0) с CLI, Web UI, тестами, ~2500 строк кода. Рабочий пайплайн: Telegram JSON → нормализация → чанкинг → атомы → retrieval → экстракция → портрет.

Реализованный стек: **regex-эвристики** (без spaCy) → **SQLite** (вместо Neo4j) + **char-TF-IDF + SVD** (вместо Qdrant). Работает полностью локально, 0 облачных зависимостей.

## 2. Сравнительный анализ по ключевым осям

### 2.1. Глубина NLP-пайплайна

| Аспект | Каркас (спецификация) | Платформа (реализация) |
|--------|----------------------|----------------------|
| Токенизация | spaCy с полной морфологией | Regex `[A-Za-zА-Яа-яЁё0-9]...` |
| Лемматизация | spaCy (морфологическая) | Суффиксное обрезание (~approximate) |
| Разбор зависимостей | Полное дерево (nsubj, obj, amod, nmod) | Нет; паттерн-матчинг по ключевым словам |
| Извлечение суждений | Субъект → Глагол → Объект из dep tree | kind + relation + concept по эвристикам |
| Модальность | 5 типов (FACTUAL → CONDITIONAL) с числовыми весами | Отсутствует |
| Качество (AFFIRM/NEG) | Из морфологии (Polarity=Neg) | Частично через keyword detection |
| Интенсивность | Лексиконы (0.3-1.0 по глаголам) | Отсутствует |
| Время | PAST/PRESENT/FUTURE из морфологии | Отсутствует |
| Пассивный залог | nsubj:pass → инверсия ролей | Не обрабатывается |
| Координация | conj → раскрытие в отдельные суждения | Не обрабатывается |
| Сложные предложения | Клаузы + модальные оболочки | Sentence split по regex |
| Дискурсивные указатели | Паттерны "проблема в том, что..." | Noise filter (промо/мусор) |
| Сравнения | Degree=Cmp → кортеж {A, B, dim, dir} | Не обрабатывается |
| Анафора | Однозначная → разрешить; неоднозначная → нет | Не обрабатывается |

**Вердикт**: Каркас описывает NLP-пайплайн уровня **серьёзного лингвистического инструмента**. Платформа реализует **прагматичный MVP** с эвристиками, сознательно отказавшимся от spaCy в пользу скорости разработки и нулевых зависимостей от тяжёлых NLP-моделей.

### 2.2. Модель данных

| Аспект | Каркас | Платформа |
|--------|--------|-----------|
| Базовая единица | Понятие-контейнер (любое значимое слово) | SemanticAtom (token + lemma + POS guess) |
| Связь | Элемент содержимого (глагол + 6 свойств) | StructuredClaim (kind + relation + concepts + domains) |
| Хранение | Neo4j (граф) + Qdrant (векторы) | SQLite (13 таблиц) |
| Граф | Направленный; A содержит B ≠ B содержит A | Плоский; claims без направленного графа |
| Грани/полисемия | DBSCAN по контекстным эмбеддингам | Не реализовано |
| PMI | log₂(P(A,B) / P(A)·P(B)), count ≥ 3 | Не реализовано |
| Конфликты | conflict=true при противоположном качестве | Не отслеживаются |
| Портрет | Не специфицирован (карта = портрет) | Развёрнутая структура: domains, stable_concepts, relations, motifs, tensions |

**Вердикт**: Каркас оперирует **микроуровнем** (каждое суждение = ребро графа с 6+ атрибутами). Платформа оперирует **макроуровнем** (claim = обобщённое высказывание с доменами и confidence). Это не противоречие, а разница в гранулярности.

### 2.3. Философские различия

Каркас и платформа разделяют один фундамент (стенографический принцип, контейнерная модель, human-in-the-loop), но **расходятся в приоритетах**:

**Каркас**: "Точность записи важнее скорости. Лучше потерять связь, чем записать неверную."
- Опирается на синтаксический разбор spaCy
- Готов принять ~5% ошибок парсера как шум
- PMI подхватит то, что синтаксис не поймал

**Платформа**: "Deterministic before probabilistic. Rule-first, model-second."
- Сознательно отказалась от spaCy ради нулевых зависимостей
- Использует keyword patterns вместо dependency parsing
- Добавила retrieval + portrait, которых нет в каркасе

## 3. Критика каждого проекта

### 3.1. Критика FVSC_логический_каркас

**Сильные стороны:**
- Исключительно глубокое теоретическое обоснование (20+ академических источников, каждое решение мотивировано)
- Стенографический принцип — подлинно оригинальный вклад, не имеющий прямых аналогов
- Элегантная формализация: суждение → ребро с 6 свойствами покрывает огромное пространство естественного языка
- Глагол как дуальная сущность — красивый синтез Davidson + FrameNet + личного опыта

**Слабости:**

1. **Нет реализации**. 1200 строк теории, 0 строк кода. Каждое архитектурное решение не прошло проверку реальными данными. Пайплайн из 7 стадий выглядит элегантно, но нет evidence что он работает на реальном чате (не на "свобода требует ответственности").

2. **Зависимость от spaCy русской модели**. Сам документ признаёт ~5% ошибок в dependency parsing для русского (LAS 95.1%). Но на реальных чатах с опечатками, сленгом, code-switching (RU/EN), неполными предложениями — ошибки будут значительно выше. Каркас не адресует degradation на messy input.

3. **Neo4j + Qdrant — тяжёлый стек**. Для персонального инструмента это overhead: требует установки, настройки, ресурсов. Противоречит духу "local-first" (хотя оба могут работать локально).

4. **Отсутствие retrieval-стратегии**. Как пользователь будет взаимодействовать с картой? Нет поиска, нет portrait, нет UI. Каркас описывает запись, но не чтение.

5. **"Я" как контейнер** — единственный открытый вопрос в документе. Но это фундаментальная проблема: бо́льшая часть чат-текстов — от первого лица. Если "я" — свалка, половина суждений теряет структуру.

6. **Нет обработки мусора**. Каркас предполагает осмысленный текст. Реальный Telegram-экспорт на 80%+ — noise: "ок", "ага", ссылки, пересылки, промо. Каркас не говорит, как с этим быть.

### 3.2. Критика platform-codex-exocortexAUTOTRY

**Сильные стороны:**
- Работает. Реальный пайплайн, проверенный на 4.4 МБ Telegram-корпусе
- Отличная система фильтрации мусора: externality scoring, materialization gate, noise penalties
- Origin tagging (direct_thought vs copypaste_external) — практичное решение проблемы, которую каркас не адресует
- Portrait как структурированный snapshot — решает проблему "как читать карту"
- Hybrid search (BM25 + TF-IDF + concept overlap) — элегантное решение retrieval без внешних API
- Web UI для инспекции — debug-first approach
- Тесты, CLI, документация — production-grade инфраструктура

**Слабости:**

1. **Экстракция примитивна по сравнению со спецификацией**. Keyword patterns ("maybe" → hypothesis, "need" → goal) не различают:
   - "Проблема в том, что нет денег" (дискурсивный указатель) от "Проблема — это сложная ситуация" (определение)
   - "Свобода требует ответственности" (утверждение) от "Свобода не требует ответственности" (отрицание)
   - Контекстуальный смысл слова от его буквального значения

2. **Потеря структуры суждения**. Каркас извлекает: кто → что → через какой глагол → с какой модальностью. Платформа извлекает: "это observation, relation=mentions, concepts=[freedom, responsibility]". Вся направленность и атрибутика суждения теряется.

3. **Approximate lemmatizer**. Суффиксное обрезание для русского — грубая операция. "Требовать" → "требов" (отрезано -ать), но "требование" → "требован" (отрезано -ие). Разные леммы для одного корня.

4. **Нет графовой структуры**. Claims — это плоские записи, не граф. Невозможно навигировать: "что содержит `свобода`? → `ответственность`, `выбор`...". Нужен JOIN по claims, а это не то же самое, что граф.

5. **Portrait слишком агрегирован**. Stable_concepts и stable_relations — это глобальная статистика. Каркас предлагает навигируемую карту (зайти в контейнер → увидеть содержимое → перейти к другому контейнеру). Портрет — это dashboard, не карта.

6. **Taxonomy hardcoded**. 13 canonical concepts, 12 relations, 7 domains — зашиты в код. Реальный пользователь оперирует сотнями понятий, десятками типов связей. Taxonomy не масштабируется.

## 4. Оценка совместимости

### 4.1. Что совместимо (полностью)

| Аспект | Совместимость | Почему |
|--------|---------------|--------|
| Философия | 100% | Одни и те же принципы: стенографический, контейнерный, local-first |
| Стек хранения | Совместим | SQLite платформы → промежуточный слой; каркас → целевой (Neo4j/Qdrant) |
| Telegram pipeline | Дополняет | Платформа обрабатывает формат; каркас обрабатывает содержание |
| Origin tagging | Дополняет | Платформа фильтрует мусор → каркас получает чистый вход |
| Portrait | Дополняет | Каркас строит карту; портрет — агрегированная проекция карты |
| Антураж | Совместим | Оба проекта выделяют роль LLM-ассистента |
| Тезаурус | Совместим | Оба проекта ссылаются на shared thesaurus layer |

### 4.2. Что несовместимо (требует reconciliation)

| Аспект | Конфликт | Решение |
|--------|----------|---------|
| Extraction engine | Regex vs spaCy dep parse | Каркас побеждает; regex — это обходной путь, не альтернатива |
| Модель claim | Flat (kind/relation/domains) vs directed (S→V→O + 6 attrs) | Модель каркаса строго мощнее; claim платформы — partial projection |
| Storage | SQLite flat tables vs граф (Neo4j) | Можно начать с SQLite + графовый view (recursive CTE или adjacency list) |
| Retrieval | char-TF-IDF+SVD vs Qdrant | Совместимы; TF-IDF — fallback; Qdrant — апгрейд |
| Единица анализа | SemanticAtom (token) vs Понятие (lemma, container) | Каркас мощнее; atom — часть понятия |

### 4.3. Что уникально в каждом проекте (и нужно обоим)

**Из каркаса в платформу нужно:**
- Dependency parsing (spaCy или Stanza) для точного извлечения суждений
- Модальность, качество, интенсивность, время как свойства ребра
- DBSCAN-кластеризация контекстов для обнаружения граней
- PMI для статистических связей
- Обработка пассива, координации, сложных предложений
- Дискурсивные указатели (не наивный noise filter)

**Из платформы в каркас нужно:**
- Origin tagging (direct_thought vs copypaste) — без этого каркас захлебнётся мусором
- Externality scoring и materialization gate
- Hybrid search и retrieval layer
- Portrait aggregation — навигационный слой поверх карты
- Web UI и CLI — пользовательский интерфейс
- Timeline events — аудит всех изменений
- Тестовая инфраструктура и стресс-тест на реальных данных

## 5. Рекомендация по объединению

### 5.1. Архитектура объединённого проекта

```
Объединённый FVSC Exocortex:

Layer 0: Ingestion (из платформы)
  ├── Telegram/Obsidian import
  ├── Origin tagging (direct_thought / copypaste / forwarded)
  ├── Externality scoring
  ├── Noise/promo filtering
  └── Message normalization

Layer 1: NLP Core (из каркаса)
  ├── spaCy dependency parsing (ru_core_news_lg)
  ├── Clause splitting (сложные предложения → клаузы)
  ├── Anaphora resolution (только однозначные)
  ├── Passive → active inversion
  ├── Coordination expansion
  ├── Discourse marker detection
  └── Compound verb unification ("привести" + "к" → "привести_к")

Layer 2: Extraction (синтез)
  ├── Judgment extraction (S → V → O + quality/modality/intensity/tense)
  │   — из каркаса
  ├── Adjective/preposition constructions (amod, nmod:от, cop:это)
  │   — из каркаса
  ├── Materialization gate (confidence threshold)
  │   — из платформы, адаптированная
  ├── Source attribution (user / citation / auto_inferred)
  │   — из каркаса
  └── Origin-based weight adjustment (copypaste × 0.62)
      — из платформы

Layer 3: Container Population (из каркаса)
  ├── get_or_create(concept, user_id)
  ├── Content element insertion (judgment → edge)
  ├── Conflict detection (AFFIRM vs NEGATIVE, same verb)
  ├── Homonym handling (Lesk via thesaurus)
  └── Context accumulation (for DBSCAN)

Layer 4: Statistical Relations (из каркаса)
  ├── PMI computation
  ├── CO_OCCURS links
  └── Promotion rules (P(B|A) - P(A|B) > 0.3)

Layer 5: Facet Discovery (из каркаса)
  ├── DBSCAN clustering of context embeddings
  ├── Facet labeling (Антураж)
  └── Homonym detection (inter-cluster sim < 0.15)

Layer 6: Retrieval (из платформы)
  ├── FTS5 (BM25)
  ├── Embeddings (TF-IDF + SVD или sentence-transformers)
  ├── Hybrid search with concept overlap
  └── Retrieval penalties (noise, promo, length)

Layer 7: Portrait & UI (из платформы)
  ├── Portrait aggregation (domains, stable_concepts, relations)
  ├── Container navigation view (из каркаса — browse what concept contains)
  ├── Web UI
  └── CLI

Layer 8: Audit & Thesaurus (синтез)
  ├── Timeline events (из платформы)
  ├── FV-Thesaurus (shared, seeded from RuWordNet etc.)
  ├── UserOverlay (personal refinements)
  └── Retrieval precedence (personal > overlay > shared > hypothesis)
```

### 5.2. Стратегия миграции

**Фаза 1: Добавить spaCy в платформу** (самый большой рычаг)
- Установить `spacy` + `ru_core_news_lg`
- Заменить regex-tokenizer на spaCy в `text.py`
- Заменить approximate lemmatizer на spaCy morphology
- Заменить keyword extraction в `extraction.py` на dependency tree traversal
- Сохранить origin tagging, materialization gate, noise filtering — они уже работают

**Фаза 2: Обогатить модель claim**
- Добавить quality (AFFIRM/NEGATIVE), modality, intensity, tense к StructuredClaim
- Добавить direction (subject → object) — claims становятся направленными
- Расширить SQLite-схему, не ломая существующую

**Фаза 3: Контейнерная навигация**
- Добавить view "что содержит понятие X" поверх claims
- Добавить conflict detection
- Добавить PMI-слой

**Фаза 4: Грани и кластеризация**
- DBSCAN по контекстным эмбеддингам
- Facet labeling
- Homonym detection

**Фаза 5: Замена хранения** (опциональная)
- SQLite → SQLite с графовым view (adjacency list) — или Neo4j если нужен
- char-TF-IDF → sentence-transformers → опционально Qdrant

### 5.3. Что НЕ нужно объединять

- **Neo4j**: SQLite + правильная схема достаточно для v1-v2. Neo4j — overhead для персонального инструмента
- **Qdrant**: аналогично. sentence-transformers + faiss/annoy хватит
- **Полную модальную систему сразу**: начать с FACTUAL vs EPISTEMIC, остальное добавить итеративно
- **Физику** (масса, упругость, аттракторы): оба проекта правильно отложили это

## 6. Общая оценка

**FVSC_логический_каркас** — это один из наиболее продуманных теоретических документов для персональной семантической системы. Глубина лингвистического обоснования, стенографический принцип и контейнерная модель — подлинно оригинальные вклады. Главная слабость: нет имплементации и нет плана работы с messy real-world data.

**platform-codex-exocortexAUTOTRY** — это зрелый MVP, который решает реальные проблемы (мусор, мультиязычность, retrieval, UI), но платит за прагматизм потерей лингвистической точности. Главная слабость: extraction engine не соответствует уровню амбиции проекта.

**Совместимость: высокая**. Проекты не конкурируют — они описывают разные слои одной системы. Каркас — это спецификация NLP-ядра. Платформа — это инфраструктура вокруг ядра. Объединение не только возможно, но и является естественным следующим шагом: вставить spaCy-based extraction из каркаса в pipeline платформы, сохранив всю обвязку (origin tagging, noise filtering, retrieval, portrait, UI).

Ключевой risk при объединении: **performance на реальных данных**. spaCy `ru_core_news_lg` на 4.4 МБ Telegram-корпусе — это ~50k+ предложений через dependency parser. Нужно профилировать и, возможно, обрабатывать инкрементально.
