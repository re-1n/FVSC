# FVSC: Логика записи персональных смыслов

## Принцип

Система — стенограф. Записывает ТОЛЬКО то, что следует из формулировок человека.
Интерпретация — после, человеком при помощи LLM (Антураж).

---

## I. Ядро: понятие как контейнер

### Центральная идея

Значение понятия = его содержимое. Не "что слово значит в словаре", а какие понятия человек помещает внутрь через свои формулировки.

Каждое понятие — одновременно контейнер и содержимое других контейнеров. Это взаимно вложенная структура: "свобода" содержит "ответственность", а "ответственность" может содержать "свободу" — но с другой гранью и другим отношением.

> Теоретическая основа: Wittgenstein (1953) — "значение слова это его употребление в языке"; Frege (1892) — различие sense/reference; Kelly (1955) — личные конструкты.
> Оригинальный вклад: модель контейнера и принцип "определение = содержимое".

### Зачем

Вопрос любой дискуссии — это вопрос определений. Люди используют одни и те же слова, но вкладывают в них разный смысл. Конфликты и непонимание возникают из-за разницы в содержимом контейнеров. Система делает эту разницу видимой.

### Пример

```
"Свобода" для человека А:
    содержит "ответственность"  (через "требовать", утвердительно, ×12)
    содержит "выбор"            (через "включать", утвердительно, ×5)
    содержит "независимость"    (через "это", утвердительно, ×3)

"Свобода" для человека Б:
    содержит "одиночество"      (через "приводить к", утвердительно, ×7)
    содержит "границы"          (через "разрушать", утвердительно, ×4)
    содержит "ответственность"  (через "не требовать", отрицательно, ×3)

Расхождение:
    А: свобода "требует" ответственности
    Б: свобода "не требует" ответственности
    → Вот источник спора. Он виден на карте.
```

## II. Все слова — понятия-контейнеры

### Не только существительные

В системе ВСЕ значимые слова — понятия-контейнеры: существительные, глаголы, прилагательные. Каждое наполняется содержимым из контекстов пользователя.

Это следует из принципа точной записи: "требовать" для разных людей значит разные вещи. Один человек наполняет "требовать" содержимым {настоять, жёсткость, ультиматум}, другой — {нуждаться, условие, необходимость}. Это различие важно и должно быть записано.

### Глагол — одновременно связка И понятие

Когда человек говорит "свобода требует ответственности":

```
1. "Требовать" как СВЯЗКА:
   Связывает "свободу" и "ответственность" в суждение.
   На ребре сохраняется: глагол="требовать", качество, модальность, интенсивность.

2. "Требовать" как ПОНЯТИЕ (контейнер):
   Само понятие "требовать" наполняется из контекстов этого человека.
   Для кого-то "требовать" содержит {настойчивость, сила, давление}.
   Для другого — {необходимость, условие, логика}.
   Это определяет, ЧТО ИМЕННО человек имеет в виду, когда говорит "требует".
```

### Академическое обоснование

Эта двойственность — не изобретение. Она хорошо обоснована в лингвистике:

**Neo-Davidsonian event semantics** (стандарт в формальной семантике): события (= глаголы) — первоклассные сущности со своими свойствами. Не просто связки, а объекты с метаданными.

**FrameNet** (Fillmore): глагол не коннектор, а эвокатор фрейма — вызывает целую концептуальную структуру. "Кусать" вызывает фрейм {агрессор, жертва, часть тела, боль}.

**VerbNet** (Levin): сотни классов глаголов, каждый с собственной семантической структурой. Исследователи прямо говорят: классификация inherently lossy — любое сведение к категориям теряет информацию.

**Cognitive Linguistics** (Langacker): значения — "ментальные процессы", не статические ярлыки. Глагол несводим к одной из 6 категорий.

**Reification** в knowledge graphs: отношения превращаются в узлы, чтобы хранить их свойства. Это именно то, что мы делаем с глаголами.

## III. Из чего состоит запись

Человеческая речь состоит из суждений — высказываний, связывающих понятия. В формальной логике:

```
СУЖДЕНИЕ = Субъект + Связка + Предикат + Свойства связки
```

### 3.1. Понятие (контейнер)

```
Понятие = {
    лемма:       "свобода"              // или "требовать", или "важный"
    user_id:     "user_123"
    содержимое:  [элементы содержимого]  // другие понятия с отношениями
    контексты:   [предложения]           // для кластеризации
    измерения:   {грани смысла}          // из собственных векторов ρ
}
```

Определение понятия для данного человека = совокупность его содержимого.

### 3.2. Элемент содержимого (суждение → ребро)

Когда человек формулирует суждение, система помещает предикат внутрь субъекта:

```
Элемент содержимого = {
    понятие:        "ответственность"   // что помещено внутрь
    связка:         "требовать"         // ТОЧНЫЙ глагол, или тип зависимости (amod, nmod)
    качество:       AFFIRMATIVE         // утвердительное или отрицательное
    модальность:    FACTUAL             // факт / возможность / необходимость / желание / условие
    интенсивность:  0.7                 // из лексической семантики связки
    время:          PRESENT             // PAST / PRESENT / FUTURE
    количество:     12                  // сколько раз встретилось
    активная_грань: 0                   // какое измерение предиката активно
    контексты:      ["свобода требует ответственности", ...]
    source:         "user"              // user / citation / user_confirmed
    conflict:       false               // противоречит ли другому элементу с тем же содержимым
    comparison:     null                // null, или {direction: ">", dimension: "важный"}
}
```

## IV. Свойства связки — что извлекается из слов

Все свойства берутся из самих формулировок. Система не интерпретирует высказывания если на это не указывают поступающие от субъекта данные.

### 4.1. Глагол (связка)

Глагол сохраняется в оригинальной лемматизированной форме. Это первичная, самая точная запись отношения.

**Для группировки и анализа** (не для записи) глаголы могут быть распределены по категориям:

```
CONTAINS  — включение          "содержать", "включать", "иметь"
REQUIRES  — необходимость      "требовать", "нуждаться", "просить"
OPPOSES   — противопоставление "противостоять", "отрицать", "разрушать"
CAUSES    — причинность        "порождать", "приводить к", "вызывать"
ENABLES   — способствование    "давать", "помогать", "открывать"
DEFINES   — определение        "являться", "это", "означать"
```

Эти категории — аналитический инструмент, а не замена глагола. При записи всегда сохраняется точный глагол.

**Неклассифицированные глаголы** не получают категорию по умолчанию. Они записываются как есть. Их категория может быть определена позже:
- Антуражем (предложение пользователю)
- Через дистрибутивное сходство (глагол появляется в тех же контекстах, что и уже классифицированные глаголы → предложить категорию)
- Через содержимое самого глагола как понятия (если "терзать" содержит {боль, мучение}, это близко к CAUSES)

**Неглагольные конструкции:**

Стенографический принцип требует: если глагола нет в тексте — не фабриковать. Вместо выдуманного глагола хранится тип зависимости или конструкция:

| Конструкция | Пример | Связка (что хранится) |
|---|---|---|
| Прилагательное-модификатор | "важная свобода" | `amod` (тип зависимости) |
| Копула "X — это Y" | "Свобода — это ответственность" | `cop:это` (копула) |
| Предлог "от" | "свобода от страха" | `nmod:от` (предлог) |
| Предлог "для" | "свобода для творчества" | `nmod:для` |
| Предлог "без" | "жизнь без свободы" | `nmod:без` |
| Предлог "с" | "свобода с ограничениями" | `nmod:с` |
| Генитив "X Y-а" | "свобода человека" | `nmod:gen` |

Хранится точная конструкция из текста, не выдуманный глагол. Это согласуется с Universal Dependencies: копула не вставляется когда её нет в оригинале.

> Источник: Universal Dependencies — copula rule: копула используется только когда она присутствует в тексте; UD Russian annotation guidelines.

### 4.2. Качество (утверждение / отрицание)

Определяется наличием отрицания при связке.

```
"Свобода требует ответственности"      → AFFIRMATIVE
"Свобода НЕ требует ответственности"   → NEGATIVE
```

Извлечение:
- Частица "не" как зависимость `advmod` при глаголе
- Морфологический признак `Polarity=Neg`
- Отрицательные местоимения: "никто", "ничто", "никогда"

Качество — свойство СВЯЗИ (суждения), не ПОНЯТИЯ. Понятие активируется при любом упоминании. "Я не свободен" — понятие "свобода" активно, связь "я → свобода" отрицательна.

> Источник: формальная логика — качество суждения (утвердительное/отрицательное); DRT (Kamp 1981) — отрицание как оператор на дискурсивных структурах.
> Оригинальный вклад: отрицание как свойство ребра, а не узла. Понятие укрепляется при любом упоминании.

### 4.3. Модальность

Определяется модальными глаголами и наклонением.

```
FACTUAL      1.0  — "свобода требует"       // изъявительное наклонение
DEONTIC      0.7  — "свобода должна"        // долженствование
DESIDERATIVE 0.6  — "я хочу свободы"        // желание
EPISTEMIC    0.5  — "свобода может"          // возможность
CONDITIONAL  0.4  — "если бы свобода..."     // условное наклонение
```

Извлечение:
- Модальные глаголы: мочь → EPISTEMIC, должен → DEONTIC, хотеть → DESIDERATIVE
- Условное наклонение: `Mood=Cond` → CONDITIONAL
- Повелительное: `Mood=Imp` → DEONTIC
- Иначе: FACTUAL

Влияние: модальность умножает вес элемента содержимого. Факт весит больше гипотезы.

> Источник: Modal DRT (Frank 1996); формальная логика — модальность суждения.

### 4.4. Интенсивность

Свойство СЛОВА, не интерпретация системы. "Ненавидеть" сильнее "не любить" в любом словаре.

```
Глаголы:
    просить      → 0.3    нуждаться    → 0.6    требовать    → 0.9
    не любить    → 0.3    ненавидеть   → 0.8    презирать    → 0.9
    нравиться    → 0.3    любить       → 0.6    обожать      → 0.9

Прилагательные:
    важный       → 0.5    необходимый  → 0.9
    хороший      → 0.4    прекрасный   → 0.8

Наречия-модификаторы:
    немного      → 0.3    очень        → 0.8    абсолютно    → 1.0
```

Интенсивность хранится на элементе содержимого.

## V. Полный пайплайн

```
ВХОД: текст от пользователя
 │
 ▼
1. СИНТАКСИЧЕСКИЙ РАЗБОР (spaCy)
 │  Токенизация → лемматизация → дерево зависимостей
 │  Определение границ клауз (для сложных предложений)
 │
 ▼
2. ПРЕДОБРАБОТКА
 │  2a. Разрешение анафоры (только однозначные случаи)
 │    Если кандидат ОДИН → разрешить
 │    Если кандидатов НЕСКОЛЬКО → НЕ разрешать, сохранить контекст
 │    Принцип: лучше потерять связь, чем записать неверную.
 │
 │  2b. Обработка пассивного залога
 │    nsubj:pass → перевернуть направление субъект/объект
 │
 │  2c. Объединение составных глаголов
 │    глагол + управляемый предлог (prep dep) → одна лемма
 │    "привести" + "к" → "привести_к"
 │
 │  2d. Раскрытие координации
 │    conj dependency → каждый элемент получает полный набор связей
 │    "свобода и ответственность требуют" → два отдельных суждения
 │
 ▼
3. КОНТЕКСТНАЯ КЛАССИФИКАЦИЯ (до извлечения суждений)
 │
 │  Определяет РЕФЕРЕНЦИАЛЬНЫЙ СТАТУС каждого NP/местоимения:
 │  о чём/ком речь в данном контексте? Это знание о грамматическом
 │  ПОВЕДЕНИИ слов, не об их значении. Стенографический принцип
 │  не нарушается: слой определяет не ЧТО человек имеет в виду,
 │  а О ЧЁМ речь (generic vs specific, self vs other).
 │
 │  3a. Классификация клаузы (уровень предложения)
 │    Тип предиката:
 │      individual-level ("быть свободным", "быть смертным") → GENERIC
 │      stage-level ("прийти", "упасть", "сказать") → EPISODIC
 │    Вид глагола (аспект):
 │      НСВ (имперфектив) → скорее GENERIC/HABITUAL
 │      СВ (перфектив) → скорее EPISODIC/SPECIFIC
 │    Наклонение:
 │      условное (Mood=Cnd) → GENERIC/CONDITIONAL
 │      повелительное (Mood=Imp) → SPECIFIC (кроме пословиц)
 │    Модальность:
 │      деонтическая ("должен") → усиливает GENERIC
 │
 │  3b. Классификация NP (уровень именной группы)
 │    Для каждого существительного/местоимения определить ref_status:
 │
 │    CONCEPTUAL — абстрактное понятие
 │      → извлекать нормально
 │      Сигналы: нет детерминера, ILP-предикат, generic/habitual clause,
 │      абстрактное существительное (свобода, любовь, истина...),
 │      нет относительной клаузы, нет числительного
 │      "Человек должен быть свободен", "Свобода — это ответственность"
 │
 │    SELF — о говорящем
 │      "я", "себя"; "я как человек..."
 │      → маршрутизировать в [self] говорящего
 │
 │    INTERLOCUTOR — о собеседнике
 │      → маршрутизировать в [self] собеседника
 │      Сигналы: "ты" + императив (Mood=Imp), "ты" + прошедшее СВ,
 │      "ты" + темпоральный маркер ("вчера", "сегодня"),
 │      "ты" в episodic clause
 │      "ты вчера сказал", "ты как?", "скажи мне"
 │
 │    GENERIC — обобщённое "ты"/"человек" = "любой"
 │      → извлекать с пониженным весом (modality *= 0.7)
 │      Сигналы: "ты" в условной клаузе ("если/когда ты..."),
 │      "ты" в generic clause (НСВ + презенс, нет темпоральных маркеров),
 │      маркеры хабитуальности ("бывает", "обычно", "всегда"),
 │      нулевое подлежащее 2-го лица ("не знаешь, что делать")
 │      Русский термин: обобщённо-личное предложение
 │      "когда ты видишь закат...", "бывает, чувствуешь..."
 │
 │    REFERENTIAL — конкретная референция
 │      → пропустить (не наполнять контейнер)
 │      Сигналы (NP-level):
 │        • Детерминер: этот/тот/мой/какой-то/другой + quasi-generic noun
 │        • Числительное: "3 человека", "несколько человек"
 │        • Имя собственное: "человек Вася"
 │        • Относительная клауза (acl:relcl): "человек, который пришёл вчера"
 │          — привязывает NP к конкретному контексту
 │      Сигналы (clause-level):
 │        • SLP-предикат + СВ (перфектив): "человек пришёл" (событие)
 │        • Темпоральный маркер: "вчера", "сегодня", "потом"
 │      Примеры:
 │        "этот человек странный" → REFERENTIAL (детерминер "этот")
 │        "человек, который младше тебя" → REFERENTIAL (relcl)
 │        "один знакомый сказал" → REFERENTIAL (числительное "один")
 │        "человек пришёл вчера" → REFERENTIAL (SLP + перфектив + "вчера")
 │
 │    QUOTE — внутри чужой речи
 │      Глагол речи + ccomp: "Кант говорил, что..."
 │      → source=citation, пониженный вес
 │
 │  3c. Лингвистический лексикон (фиксированный, из лингвистики)
 │    Не тезаурус значений (это строят ρ), а тезаурус ПОВЕДЕНИЯ:
 │    • Тип предиката: ILP vs SLP
 │    • Тип глагола: модальный / речевой / ментальный / действие
 │    • Поведение существительного: abstract / quasi-generic / discourse-pointer
 │    • Маркеры обобщённости: "бывает", "случается", "обычно"
 │
 │  > Источники: ACE Entity Class System (LDC 2005) — SPC/GEN/USP;
 │  > Reiter & Frank 2010 (ACL) — Identifying Generic NPs;
 │  > Friedrich & Pinkal 2015 (ACL) — clause-level genericity;
 │  > Krifka et al. 1995, "The Generic Book" — ILP vs SLP;
 │  > Обобщённо-личные предложения (русская грамматическая традиция);
 │  > Derczynski et al. 2015 — impersonal uses of 2nd person in Russian.

 │  3a. Главный алгоритм: walk(node, ctx, results)
 │
 │    Для каждого предложения:
 │      root = sent.root
 │      ctx = DEFAULT_CONTEXT  (negated=false, modality=1.0, ...)
 │      walk(root, ctx, results)
 │
 │    walk(node, ctx, results):
 │      1. ctx = apply_context_modifiers(node, ctx)
 │         → обнаружить отрицание, кванторы, морфологию на ЭТОМ узле
 │      2. Классифицировать узел:
 │         CLAUSE_HEAD (глагол) | CONJUNCTION (conj)
 │         | NOMINAL_PREDICATE (копула) | AMOD (прилагательное)
 │         | OTHER (рекурсия в потомков)
 │      3. Диспетчеризация по типу
 │      4. Рекурсия в подчинённые клаузы (ccomp, xcomp, advcl, acl)
 │
 │    Контекст передаётся ТОЛЬКО вниз. Каждый рекурсивный вызов
 │    может модифицировать ctx для потомков, но не для родителей.
 │    Сложность: O(N) — каждый токен посещается ровно один раз.
 │
 │  3b. Скоупинг отрицания
 │
 │    Отрицание — не бинарный флаг на глаголе, а СКОУП в дереве.
 │    "Не" при глаголе → toggle ctx.negated для его поддерева.
 │
 │    Neg-raising глаголы (думать, считать, полагать):
 │      "Я НЕ думаю, что свобода требует" →
 │        отрицание на "думать" ПРОБРАСЫВАЕТСЯ во внутреннюю клаузу
 │        → свобода→требовать→ответственность получает NEGATIVE
 │
 │    Не-neg-raising (знать, бояться):
 │      "Я НЕ знаю, что свобода требует" →
 │        отрицание ОСТАЁТСЯ на оболочке
 │        → свобода→требовать→ответственность получает EPISTEMIC(0.5)
 │
 │    Двойное отрицание: toggle + toggle = AFFIRMATIVE
 │      "Не мог не видеть" → видеть = AFFIRMATIVE, модальность DEONTIC
 │
 │  3c. Модальные оболочки
 │
 │    Глаголы-оболочки (верить, думать, хотеть) + ccomp/xcomp:
 │    → НЕ извлекать суждение из оболочки
 │    → Передать модальность во внутреннюю клаузу
 │
 │    Маппинг:
 │      знать      → FACTUAL(1.0)      бояться     → EPISTEMIC(0.4)
 │      верить     → EPISTEMIC(0.5)    сомневаться → EPISTEMIC(0.3)
 │      думать     → EPISTEMIC(0.5)*   надеяться   → DESIDERATIVE(0.5)
 │      считать    → EPISTEMIC(0.6)*   хотеть      → DESIDERATIVE(0.6)
 │      полагать   → EPISTEMIC(0.5)*   мечтать     → DESIDERATIVE(0.5)
 │
 │      * = neg-raising (пробрасывает отрицание вниз)
 │
 │    Если внутренняя клауза уже имеет свою модальность,
 │    модальности перемножаются: итоговая = outer × inner.
 │
 │  3d. Условные конструкции
 │
 │    Детекция: mark(clause, "если"/"когда"/"раз") при dep_="advcl"
 │    Или: морфология Mood=Cnd (условное наклонение)
 │
 │    "Если свобода есть, то ответственность неизбежна":
 │      → Антецедент + Консеквент получают один condition_id
 │      → Оба получают modality = CONDITIONAL(0.4)
 │      → Расширение суждения: condition_id, condition_role
 │
 │    Если антецедент позже подтверждён ("свобода есть"),
 │    консеквент может быть промотирован до FACTUAL.
 │
 │  3e. Кванторы
 │
 │    Кванторы модифицируют вес суждения через quant_weight:
 │
 │    Универсальные (усиление):
 │      все/весь → ×1.3    каждый → ×1.4    любой → ×1.2
 │      всегда → ×1.3     (временной универсальный)
 │
 │    Экзистенциальные (ослабление):
 │      некоторый → ×0.7   какой-то → ×0.6   один → ×0.5
 │      иногда → ×0.6     (временной экзистенциальный)
 │
 │    Итоговый вес: w = modality × intensity × quant_weight
 │    "Все друзья любят X" весит больше чем "один друг любит X"
 │
 │  3f. Координация (раскрытие)
 │
 │    expand_coordination(node) → раскрыть conj-зависимости в список
 │    Декартово произведение субъектов × объектов:
 │
 │    "Свобода и ответственность требуют мужества и честности"
 │    → 2 × 2 = 4 суждения:
 │      свобода→требовать→мужество, свобода→требовать→честность
 │      ответственность→требовать→мужество, ответственность→требовать→честность
 │
 │  3g. Прилагательные (amod dependency):
 │    существительное = контейнер
 │    прилагательное = содержимое (связка = "amod")
 │    Если advmod ("очень важный"):
 │      интенсивность × модификатор ("немного" ×0.5, "очень" ×1.3)
 │
 │  3h. Неглагольные конструкции:
 │    копула → "cop:это", предлоги → "nmod:предлог", генитив → "nmod:gen"
 │    Хранится точная конструкция, не выдуманный глагол
 │
 │  3i. Дискурсивные указатели:
 │    "проблема в том, что..." → НЕ наполнять "проблему"
 │    Содержимое направить в реальные понятия
 │
 │  3j. Сравнения (Degree=Cmp): записать как comparison с direction >/<
 │
 │  3k. Безличные предложения: дативный субъект → использовать;
 │    нет субъекта → суждение не извлекается, контекст сохраняется
 │
 │  3l. Восклицания без предиката:
 │    Суждение не извлекается. Понятие активируется, контекст сохраняется.
 │
 │  3m. Глагол как понятие:
 │    get_or_create(глагол_лемма, user_id)
 │    → глагол тоже контейнер, наполняемый из контекстов
 │
 │  ┌─────────────────────────────────────────────────┐
 │  │  FLAT vs RECURSIVE — сравнение на примерах       │
 │  │                                                 │
 │  │  "Я не думаю, что свобода требует жертв":       │
 │  │    FLAT:  своб→треб→жертва, AFFIRM, mod=1.0     │
 │  │    RECUR: своб→треб→жертва, NEGATIVE, mod=0.5   │
 │  │                                                 │
 │  │  "Если свободен — несёшь ответственность":      │
 │  │    FLAT:  два несвязанных суждения, mod=1.0      │
 │  │    RECUR: два суждения с condition_id, mod=0.4   │
 │  │                                                 │
 │  │  "Все друзья считают, что любовь требует свободы":│
 │  │    FLAT:  люб→треб→своб, AFFIRM, mod=1.0        │
 │  │    RECUR: люб→треб→своб, AFFIRM, mod=0.78       │
 │  │           (0.6 × 1.3 = epistemic × universal)   │
 │  │                                                 │
 │  │  "Свобода и ответственность требуют мужества":  │
 │  │    FLAT:  1 суждение (только первый nsubj)       │
 │  │    RECUR: 2 суждения (раскрытие координации)     │
 │  └─────────────────────────────────────────────────┘
 │
 ▼
5. МАТЕРИАЛИЗАЦИЯ В ГИЛЬБЕРТОВОМ ПРОСТРАНСТВЕ
 │  Для каждого извлечённого суждения S → V → O:
 │
 │  5a. Структура суждения (Judgment):
 │    subject:        str         // лемма субъекта
 │    verb:           str         // лемма глагола (или "amod", "cop:это", "nmod:gen")
 │    object:         str         // лемма объекта
 │    quality:        AFFIRMATIVE | NEGATIVE
 │    modality:       float       // 0.0–1.0+ (из контекстного классификатора)
 │    intensity:      float       // 0.0–1.0 (из лексикона интенсивности)
 │    timestamp:      float       // epoch seconds
 │    source_text:    str         // оригинальный текст
 │    condition_id:   int?        // связка if→then (общий id для пары)
 │    condition_role: str?        // "ANTECEDENT" | "CONSEQUENT"
 │
 │  5b. Построение вектора суждения (АСИММЕТРИЧНОЕ):
 │
 │    Для СУБЪЕКТА (вектор помещается в ρ субъекта):
 │      v_obj  = role_transform(get_vector(O), "object_in_subject")
 │      v_verb = role_transform(get_vector(V), "verb_in_subject")
 │      sign   = −1 if NEGATIVE else +1
 │      |v⟩    = normalize(v_obj + sign · intensity · v_verb)
 │
 │    Для ОБЪЕКТА (вектор помещается в ρ объекта):
 │      v_subj = role_transform(get_vector(S), "subject_in_object")
 │      v_verb = role_transform(get_vector(V), "verb_in_object")
 │      sign   = −1 if NEGATIVE else +1
 │      |v'⟩   = normalize(v_subj + sign · intensity · v_verb)
 │
 │    role_transform(v, role): детерминированная ротация
 │      Q, _ = QR(random_matrix(seed=hash(role)))
 │      return normalize(Q · v)
 │
 │    Разные роли → разные ротации → одно слово в позиции субъекта
 │    и объекта даёт РАЗНЫЕ вектора. Это обеспечивает асимметрию.
 │
 │    v1 (текущая реализация): формулы выше (сложение + QR-ротация).
 │    v2 (цель): DisCoCat тензорная свёртка v_j = Σᵢₖ subj_i · T_ijk · obj_k.
 │
 │  5c. Вычисление веса и распределение:
 │
 │    w = modality × intensity
 │    (modality уже включает quant_weight из контекстного классификатора)
 │
 │    Распределение по контейнерам:
 │      ρ(S) += w     · |v⟩⟨v|    // субъект: полный вес
 │      ρ(O) += w×0.5 · |v'⟩⟨v'|  // объект: половинный вес
 │      ρ(V) += w×0.2 · |v''⟩⟨v''| // глагол-контейнер: пятая часть
 │
 │    Соотношение 1.0 : 0.5 : 0.2 отражает:
 │      - субъект определяется тем, что говорящий помещает внутрь (полный вес)
 │      - объект активируется, но не определяется этим суждением (половина)
 │      - глагол обогащается контекстом, но медленнее (пятая часть)
 │
 │  5d. Маршрутизация "я" ([self] контейнер):
 │
 │    Если subject = "я" или "себя":
 │      self_type = classify(verb):
 │        identity ("являться", "быть", "стать"...): → [self], вес = w
 │        relation ("хотеть", "любить", "бояться"...): → [self], вес = w × 0.5
 │        action (все остальные глаголы): → НЕ в [self]
 │      Объект всегда активируется: ρ(O) += w×0.5 · |v'⟩⟨v'|
 │
 │    Соотношение identity:relation = 1:0.5 —
 │      "я являюсь свободным" весит вдвое больше чем "я хочу свободу"
 │      (кто я > что я хочу, в контексте самоидентификации)
 │
 │  5e. Provenance:
 │    log_component(S, |v⟩, w, judgment_text, timestamp)
 │
 │  5f. Conflict detection:
 │    Если в ρ(S) уже есть компонент с тем же V и O,
 │    но противоположным качеством → conflict = true
 │
 │  5g. Омонимия: если тезаурус знает омонимы →
 │    сопоставить контекст с тезаурусными ρ (через trace inner product)
 │
 ▼
6. ОБНАРУЖЕНИЕ ГРАНЕЙ (собственные вектора)
 │  Для каждого понятия с ≥ 4 компонентами в ρ:
 │    eigendecomposition(ρ) → eigenvectors (грани) + eigenvalues (веса)
 │    Каждый собственный вектор = одна грань смысла
 │    Количество значимых граней = eigenvalues > threshold
 │
 │  Полисемия: S(ρ) = -Tr(ρ · log(ρ)) — энтропия фон Неймана
 │    S > 0 → понятие полисемично
 │
 │  Омонимия: если ортогональные собственные вектора
 │    (inter-eigenvector similarity ≈ 0) → флаг potential_homonym
 │
 ▼
7. РЕКУРСИВНОЕ УГЛУБЛЕНИЕ (фрактальная структура)
 │  После прямой материализации (шаг 4) — итеративное углубление:
 │
 │  Для k = 1, 2, ..., K (или до сходимости):
 │    Для каждого понятия A:
 │      содержимое = {(Bᵢ, rᵢ, wᵢ) : contain(A→Bᵢ) > threshold}
 │      ρₖ(A) = α · ρ_direct(A) + (1-α) · Σᵢ wᵢ · transform(ρₖ₋₁(Bᵢ), rᵢ)
 │
 │  Каждая итерация углубляет представление:
 │    k=0 — только прямые суждения
 │    k=1 — + содержимое первого уровня
 │    k=2 — + содержимое содержимого
 │    ...
 │  Сходимость гарантирована: (1-α) < 1 обеспечивает затухание
 │
 │  Результат: ρ(A) — фрактальный контейнер, содержащий другие
 │  контейнеры, содержащие контейнеры, на любую глубину
 │
 ▼
8. СТАТИСТИЧЕСКИЕ СВЯЗИ (PMI)
 │  Для пар понятий, не связанных синтаксически:
 │    PMI(A,B) = log₂(P(A,B) / (P(A)·P(B)))
 │    Если PMI > 0 и count ≥ 3 → обновить ρ(A) и ρ(B) соответственно
 │
 │  Промоция в содержимое:
 │    Если contain(A→B) > 0.3 (из trace inner product) → A содержит B
 │
 │  Имплицитные связи: ко-окуренция на уровне абзаца
 │    с пониженным весом → PMI подхватит при повторении
 │
 ▼
9. ГЕНЕРАЦИЯ ГРАФОВОГО VIEW И СОХРАНЕНИЕ
 │  7a. Графовый view:
 │    Для каждой пары (A, B): contain(A→B) = Tr(ρ_A · ρ_B) / Tr(ρ_A)
 │    Если contain > threshold → ребро A→B
 │    Свойства ребра: из компонентов (глагол, качество, модальность...)
 │
 │  7b. Сохранение:
 │    Операторы плотности (low-rank: top-k eigenvectors)
 │    Компоненты с provenance (суждение, вес, timestamp)
 │    Оригинальные тексты (для полного исторического контекста)
 │    Графовый view (кэш, регенерируемый из ρ)
```

## VI. Производные свойства (вычисляются, не записываются)

Эти свойства следуют из содержимого:

```
Масса понятия = количество элементов содержимого + количество вхождений
               в другие контейнеры.

Вес элемента = количество × модальность × интенсивность × recency
               Определяет, насколько централен этот элемент для определения.

  recency = exp(-λ × (now - last_timestamp))
  λ — коэффициент затухания (конфигурируемый)
  Недавние элементы весят больше старых, но старые не исчезают.

Определение = содержимое, отсортированное по весу.
               Первые 3-5 элементов — ядро определения.

Delta = разница между содержимым у пользователя и содержимым в тезаурусе.
```

> Источник: экспоненциальное затухание — стандарт в IR и рекомендательных системах; half-life decay model (CEUR Workshop 2038).

## VII. Грани смысла и омонимия

### Полисемия (грани одного понятия)

Одно слово может нести разные значения для одного человека.

Грани обнаруживаются через **собственные вектора оператора плотности** (см. 13.5):

```
eigendecomposition(ρ_свобода) → {(λ₁, v₁), (λ₂, v₂), ...}

v₁ — главная грань (наибольшее собственное значение λ₁)
v₂ — вторая грань
...

Количество граней = количество собственных значений > порога
```

Мера полисемичности — энтропия фон Неймана: S(ρ) = -Σᵢ λᵢ · log(λᵢ). Если S ≈ 0, понятие однозначное (одна грань). Если S > 0, понятие полисемичное.

Когда "любовь" содержит "свободу", активна конкретная грань "свободы":

```
"любовь" содержит "свободу" (грань v₁: личное пространство)
"политика" содержит "свободу" (грань v₂: права и законы)

Какая грань активна — определяется через проекцию вектора суждения
на собственные вектора ρ.
```

Ядро обнаруживает грани автоматически (из собственных векторов ρ). Маркировку может делать Антураж или сам пользователь.

### Омонимия (разные понятия за одной формой)

"Ключ" (от двери) и "ключ" (родник) — не грани одного понятия, а разные слова в одной форме. Они должны быть разными контейнерами.

**Три механизма разделения:**

**1. Тезаурус как стартовый разделитель.**

Тезаурус уже знает известные омонимы:
```
ключ_1 (L0): содержит {замок, дверь, открывать, металл}
ключ_2 (L0): содержит {вода, источник, бить, родник}
```

При встрече "ключ" в тексте — сопоставить контекст с содержимым тезаурусных контейнеров:
```
"я нашёл ключ в кармане"
  контекст: {найти, карман}
  → ближе к ключ_1 {замок, дверь} → наполнить ключ_1
```

Это алгоритм Леска (Lesk 1986) — структурное сопоставление, не интерпретация.

**2. Ортогональные собственные вектора как детектор неизвестных омонимов.**

Если собственные вектора ρ почти ортогональны друг другу — грани не связаны, это разные понятия:
```
"мышка" — собственный вектор v₁: {компьютер, клик, курсор}
          собственный вектор v₂: {серая, хвост, кошка}
          ⟨v₁|v₂⟩ ≈ 0.05

→ Флаг: potential_homonym = true
```

Порог: inter-eigenvector similarity < 0.15 → вероятный омоним.

**3. Пользователь/Антураж подтверждает.**

Разделение контейнера на два — это структурное решение. Ядро обнаруживает и предлагает, но не выполняет без подтверждения:
```
Антураж: "Ты используешь 'ключ' в двух совершенно разных контекстах:
          про двери и про воду. Это для тебя два разных понятия?"
Пользователь: "Да"
→ Разделить на ключ_1 и ключ_2, раскидать контексты
```

### Ключевой принцип: карта отражает восприятие, не словарь

Граница между полисемией и омонимией размыта. Люди используют язык как инструмент и часто наполняют слова своим смыслом: видят связь там, где тезаурус видит два отдельных понятия, или наоборот — разделяют то, что тезаурус считает одним словом.

Карта записывает восприятие ЭТОГО человека:
- Если для человека "ключ" (дверной) и "ключ" (родник) — одно понятие с двумя гранями → пусть так
- Если для человека "свобода" — два совершенно разных понятия (политическая и личная) → пусть разделятся
- Тезаурус подсказывает, но не диктует

## VIII. Дискурсивные указатели (локальное vs глобальное)

### Проблема

Когда человек говорит "проблема в том, что нет денег" — слово "проблема" не определяется. Оно используется как указатель: "вот на что я обращаю внимание". Если система наивно запишет "проблема содержит деньги" — она загрязнит глобальное понятие конкретикой из одного диалога.

### Два типа использования слова

**Концептуальное** — человек определяет или связывает понятие:
- "Проблема — это сложная ситуация" → определяет. Записать.
- "Проблемы закаляют характер" → связывает. Записать.

**Референциальное** — слово служит указателем на конкретную ситуацию:
- "Проблема в том, что нет денег" → указывает. НЕ записывать как определение.
- "Моя цель — стать свободным" → указывает. НЕ записывать "цель содержит свободу".

### Какие слова ведут себя так

Класс абстрактных существительных, которые часто работают как дискурсивные указатели:

```
проблема, вопрос, суть, дело, причина, цель, задача,
смысл, итог, вывод, результат, следствие, условие,
основа, главное, важное, интересное
```

### Детекция: паттерны указательного использования

```
"{слово} в том, что ..."        → указатель
"{слово} заключается в ..."     → указатель
"{слово} состоит в ..."         → указатель
"вот {слово}: ..."              → указатель
"вся {слово} в ..."             → указатель
"эта/наша/моя {слово} — ..."   → указатель (определитель + абстрактное)
```

Если паттерн не совпадает → концептуальное использование → записывать нормально.

### Что делать при обнаружении указателя

```
"Проблема в том, что нет денег"

НЕ записывать: "проблема" содержит "деньги"

Записать:
  - Контекст сохраняется целиком (для кластеризации "проблемы" как понятия)
  - Содержимое направляется в реальные понятия из предложения:
    → "деньги" получает контекст с отрицанием
  - Пометка: scope=local (это про конкретную ситуацию, не про понятие)
```

### Академическое обоснование

- **DRT (Kamp 1981)**: discourse referent — временная сущность внутри дискурса, не концепт
- **Centering Theory (Grosz, Joshi, Weinstein 1995)**: отслеживание центра внимания в диалоге
- **Information Structure (Lambrecht 1994)**: topic (о чём речь) vs focus (что нового). Указатель — topic-маркер, реальное содержимое — focus

## X. Тезаурус (базовый слой)

Тезаурус — строительные леса. Даёт начальные определения, которые замещаются персональными.

```
Фаза 0 (нет данных):   тезаурус = единственный источник определений
Фаза 1 (1-50 текстов): персональное содержимое начинает накапливаться
Фаза 2 (50-200):       гибрид
Фаза 3 (200+):         персональная структура доминирует
```

Три функции:
1. **Начальные определения** — пока человек мало написал (временно)
2. **Эталон для сравнения** — delta показывает отклонение от нормы (постоянно)
3. **Словарный каркас** — начальные связи (транзиционно)

> Источник: Osgood (1957) — измерение отклонения личного значения от нормы; Frege (1892) — reference как общий базовый слой.
> Техника: алгоритм Леска (1986) для разрешения омонимии через тезаурус.

## XIII. Решённые вопросы

### 13.1. Сложные предложения с вложенными клаузами

Принцип: разбить на клаузы, каждую обработать отдельно, связь между клаузами определяет модальность или отношение.

**Модальные оболочки** — глагол внешней клаузы задаёт модальность внутренней:

```
"Я верю, что свобода требует ответственности"

  Записать:
    "свобода" содержит "ответственность"
    (через "требовать", AFFIRMATIVE, модальность=EPISTEMIC)

  "Верю" — модальная оболочка, не отдельное суждение о свободе.
  Оригинальный текст сохраняется целиком в контексте.
```

Маппинг глаголов-оболочек:
```
"знать, что..."          → FACTUAL
"верить, что..."         → EPISTEMIC
"думать, что..."         → EPISTEMIC
"хотеть, чтобы..."      → DESIDERATIVE
"сомневаться, что..."    → EPISTEMIC + NEGATIVE
"бояться, что..."        → EPISTEMIC (контекст сохранит окраску)
```

**Причинные, условные, уступительные** — два суждения, союз определяет связь:

```
"Свобода важна, потому что она даёт выбор"
  Суждение 1: "свобода" — "важный" (через "являться", AFFIRMATIVE)
  Суждение 2: "свобода" содержит "выбор" (через "давать", AFFIRMATIVE)
  Связь: "потому что" → суждение 2 — причина суждения 1

"Если свобода есть, то ответственность неизбежна"
  Суждение 1: "свобода" (CONDITIONAL)
  Суждение 2: "ответственность" — "неизбежный" (CONDITIONAL)
  Оба суждения получают модальность CONDITIONAL

"Свобода важна, хотя и опасна"
  Суждение 1: "свобода" — "важный" (AFFIRMATIVE)
  Суждение 2: "свобода" — "опасный" (AFFIRMATIVE)
  Оба записываются. "Хотя" не отменяет ни одно.
```

**Координация** — независимые суждения:
```
"Свобода требует ответственности и порождает страх"
  Суждение 1: "свобода" содержит "ответственность" (через "требовать")
  Суждение 2: "свобода" содержит "страх" (через "порождать")
```

> Источник: DRT (Kamp 1981) — обработка сложных дискурсивных структур; RST (Mann & Thompson 1988) — риторические отношения между клаузами.

### 13.2. Вопросы

```
Риторический вопрос = утверждение с модальностью EPISTEMIC.
  "Разве свобода не требует ответственности?"
  → "свобода" содержит "ответственность"
    (через "требовать", AFFIRMATIVE, EPISTEMIC)
  Детекция: "разве", "неужели", "не ... ли" + содержательный предикат.

Прямой вопрос = запрос без предиката.
  "Что такое свобода?"
  → Суждение НЕ извлекается. Контекст сохраняется.
  → Антураж может заметить: "ты спрашивал, но не определил".
```

> Источник: Bhattasali & Cytryn (ACL) — автоматическая детекция риторических вопросов через SVM и n-gram модели; для русского языка синтаксические маркеры ("разве", "неужели", "не...ли") — наиболее надёжные индикаторы.

### 13.3. Цитирование

```
"Кант говорил, что свобода — это ответственность"

  Записать:
    "свобода" содержит "ответственность"
    (через "являться", AFFIRMATIVE, FACTUAL)
    source = citation, author = "Кант"

  source=citation → вес ниже чем source=user.
  Но факт ВЫБОРА цитаты сохраняется — человек выбрал именно эту,
  а это говорит о его восприятии.

  Детекция: глаголы речи ("говорить", "утверждать", "писать",
  "считать") + именованная сущность или "кто-то".
```

> Источник: Attribution relation в анализе дискурса — разделение frame ("он сказал") и report (содержимое); Aikhenvald — теория evidentiality (маркирование источника информации: прямое восприятие / чужие слова / умозаключение).

### 13.4. Минимальный корпус

```
Отдельное понятие:
  ≥ 3 контекста — первое содержимое осмысленно
  ≥ 5 — содержимое отражает паттерн
  ≥ 8-10 — возможна кластеризация (обнаружение граней)

Карта в целом:
  ~30 текстов — первые устойчивые связи
  ~50 текстов (~200 контекстов) — PMI статистически значим
  ~200 текстов — надёжные грани и кросс-связи

Доверие:
  unit_confidence = min(1.0, количество_контекстов / 10)
  map_confidence = min(1.0, всего_контекстов / 200)
```

> Источник: Church & Hanks (1990) — PMI, принцип "quality > quantity"; PPMI (Positive PMI) — решение ненадёжности отрицательных значений при малом корпусе; DBSCAN — рекомендация MinPts ≥ D+1 (scikit-learn). Жёсткого минимума в литературе нет — рекомендуется байесовское сглаживание + мониторинг стабильности.

### 13.5. Гильбертово пространство — оператор плотности как структурный примитив

Вектор — не служебный инструмент, а **единственное структурное представление**. Каждое понятие — оператор плотности (density matrix) в Гильбертовом пространстве. Граф, portrait, retrieval — навигационные проекции (views) поверх этого пространства.

#### Почему оператор плотности, а не стандартный эмбеддинг

Стандартный эмбеддинг (точка в R^n) не может быть контейнером: точка ничего не содержит. Box embeddings (гиперпрямоугольники) решают containment, но не поддерживают **взаимную асимметричную вложенность** — ядро контейнерной модели FVSC:

```
"любовь" содержит "свободу" на 70%
"свобода" содержит "любовь" на 40%
```

В box embeddings это геометрически невозможно (box A внутри box B исключает обратное). Та же проблема у гиперболических конусов.

Оператор плотности решает это естественно.

#### Формализация

Понятие = оператор плотности ρ (положительно полуопределённая матрица, trace ≥ 0) в Гильбертовом пространстве H = R^d.

```
ρ(понятие) = Σᵢ wᵢ · |vᵢ⟩⟨vᵢ|
```

Где:
- |vᵢ⟩ — вектор i-го суждения, помещённого в данный контейнер
- wᵢ — вес суждения (из количества, модальности, интенсивности, recency)
- |vᵢ⟩⟨vᵢ| — rank-1 проектор (внешнее произведение вектора на себя)

Каждый компонент |vᵢ⟩⟨vᵢ| — это **буквально суждение внутри контейнера**. Оператор плотности — это контейнер, составленный из суждений.

#### Содержание (containment) — непрерывное и асимметричное

```
contain(A → B) = Tr(ρ_A · ρ_B) / Tr(ρ_A)
```

Эта мера:
- **Асимметрична**: contain(A→B) ≠ contain(B→A), потому что Tr(ρ_A) ≠ Tr(ρ_B)
- **Непрерывна**: число от 0 до 1, не бинарное "да/нет"
- **Взаимна**: оба направления определены одновременно

Более строгая мера содержания — **мера Лёвнера** (Bankova, Coecke, Lewis, Marsden, 2019):

```
hyp(A, B) = 1 - Σ_{λᵢ < 0} |λᵢ| / Tr(A)
где λᵢ — собственные значения матрицы (B - A)
```

Это graded hyponymy: "насколько A является подмножеством B" — непрерывная версия ⊆.

#### Грани смысла — собственные вектора

Полисемия обнаруживается без внешней кластеризации (DBSCAN). Собственные вектора ρ — это грани смысла:

```
eigendecomposition(ρ_свобода) → {(λ₁, v₁), (λ₂, v₂), ...}

v₁ — главная грань (наибольшее собственное значение λ₁)
v₂ — вторая грань
...
```

Количество значимых граней = количество собственных значений выше порога.

Мера полисемичности — энтропия фон Неймана:
```
S(ρ) = -Tr(ρ · log(ρ)) = -Σᵢ λᵢ · log(λᵢ)
S = 0  → однозначное понятие (одна грань)
S > 0  → полисемичное (несколько граней)
```

Мера чистоты (обратная полисемии):
```
γ(ρ) = Tr(ρ²)
γ = 1    → полностью однозначное
γ = 1/d  → максимально полисемичное
```

#### Детерминированное формирование — два уровня

Оператор плотности конструируется в два этапа. Оба детерминированы.

**Этап 1 — Прямая материализация (из суждений):**

Для каждого суждения S → V → O:
```
1. |v⟩ = encode(S, V, O)

   v2 (цель): тензорное сворачивание (DisCoCat, Coecke et al. 2010):
     v_j = Σᵢₖ subj_i · T_ijk · obj_k
     где T — тензор глагола (seed из тезауруса)

   v1 (текущая реализация): упрощённое кодирование:
     v = v_obj + intensity · v_verb
     с role-dependent QR-ротацией для асимметрии subject/object
     Достаточно для proof-of-concept; тензорная свёртка — цель v2.

2. w = count × modality_weight × intensity × quant_weight × recency
   (quant_weight — новый множитель от квантора, см. шаг 3e)

3. ρ₀(S) += w · |v⟩⟨v|   // начальный ρ из прямых суждений
```

На этом этапе ρ₀ — это "плоское" представление: контейнер содержит суждения-вектора, но не другие контейнеры.

**Этап 2 — Рекурсивное углубление (ρ из ρ):**

Ключевая идея: оператор плотности понятия определяется **из операторов плотности его содержимого**, а не только из суждений-векторов. Содержимое — тоже операторы плотности, определяемые из своего содержимого. Это фрактальная, самоподобная структура.

```
ρₖ₊₁(A) = α · ρ_direct(A) + (1-α) · Σᵢ wᵢ · transform(ρₖ(Bᵢ), rᵢ)

где:
  ρ_direct(A)     — прямые компоненты из суждений (этап 1, неизменен)
  Bᵢ              — понятия, содержащиеся в A
  rᵢ              — отношение (глагол + свойства), через которое Bᵢ входит в A
  transform(ρ, r) — преобразование ρ содержимого с учётом отношения
  α               — баланс между прямыми суждениями и содержимым (0 < α < 1)
  k               — номер итерации
```

Каждая итерация "углубляет" представление:
- k=0: ρ₀ отражает только прямые суждения о понятии
- k=1: ρ₁ отражает прямые суждения + содержимое первого уровня
- k=2: ρ₂ отражает + содержимое содержимого
- ...
- k→∞: ρ сходится к неподвижной точке, отражающей всю глубину фрактала

Итерации сходятся (аналогично PageRank или message passing в GNN), потому что коэффициент (1-α) < 1 обеспечивает затухание на каждом уровне вложенности.

**Это и есть фрактальная структура**: каждый ρ содержит другие ρ, которые содержат другие ρ. Один тип объекта на каждом уровне. Содержимое определяет форму контейнера. Контейнер может содержать контейнеры, которые содержат его — взаимная вложенность через разные грани, с разными весами, без геометрического противоречия.

**Детерминизм**: одни суждения + одно k итераций = один результат. Система использует рекурсивную алгебраическую архитектуру, структурно изоморфную TreeRNN/GNN, но с детерминированными операциями (см. раздел XIV). Нейросеть не участвует — параметры не обучаются, а накапливаются.

**Provenance**: прямые компоненты (ρ_direct) трассируются до конкретных суждений с timestamp. Рекурсивные компоненты трассируются через цепочку содержания: "ρ(свобода) включает ρ(ответственность) потому что суждение X, а ρ(ответственность) включает ρ(долг) потому что суждение Y".

#### Граф как view

```
Для каждой пары понятий (A, B):
  c = contain(A → B)
  Если c > threshold → ребро A → B в графе
  Свойства ребра (глагол, качество, модальность) — из компонентов ρ

Граф = проекция пространства, не source of truth.
```

#### Retrieval — через trace inner product

```
Sim(A, B) = Tr(ρ_A · ρ_B)
```

Это мера близости, учитывающая не только позицию (как cosine similarity), но и **форму** понятия (его грани, размах, содержимое). Два понятия с одинаковым содержимым будут иметь высокий trace inner product.

#### Практические параметры

```
Размерность:          d = 50–100 (v1)
Хранение (full):      d² floats = 80 KB при d=100
Хранение (low-rank):  r×d floats ≈ 8 KB при d=100, r=10
10,000 понятий:       ~80 MB (low-rank) — trivial для local-first
Trace inner product:   O(d²) ≈ 1 мкс на CPU
Eigendecomposition:    O(d³) ≈ 100 мкс на CPU
GPU:                   не нужен. NumPy + SciPy достаточно.
```

#### Множественность пространств — тензорное произведение

Оператор плотности может жить в тензорном произведении пространств:
```
ρ ∈ H_semantic ⊗ H_relational ⊗ H_contextual
```
- H_semantic — семантическое содержимое
- H_relational — тип отношения (глагол)
- H_contextual — модальность, качество, время

Partial trace по одному из пространств даёт проекцию: "покажи только семантику, без учёта модальности".

Это реализует множественность пространств: не одно плоское пространство, а структурированная комбинация.

> Теоретический фундамент: Coecke, Sadrzadeh, Clark (2010) — DisCoCat; Bankova, Coecke, Lewis, Marsden (2019) — Graded Hyponymy через density matrices; Piedeleu et al. (2015) — density matrices для полисемии; Meyer & Lewis (2020) — Modelling Lexical Ambiguity with Density Matrices; Coecke (2020) — DisCoCirc (arXiv:2007.03834) — text-level композиция через completely positive maps; Van Rijsbergen (2004) — The Geometry of Information Retrieval.
> Оригинальный вклад: применение density matrix semantics к персональной семантической карте; containment как trace inner product для контейнерной модели; грани как собственные вектора оператора плотности.

### 13.6. Физика (упругость, аттракторы)

```
Фундамент: контейнеры и их содержимое. Работает без физики.

Физика — опциональная надстройка для фазы 2+:
  - Масса = количество содержимого (уже следует из данных)
  - Упругость = сопротивление изменению (полезно для "старых" понятий)
  - Равновесие = каскадное обновление (при большом импорте)
  - Аттракторы = устойчивые состояния (для визуализации)

Сначала проверить фундамент. Физику добавить когда контейнерная
модель работает.
```

### 13.7. Зачем карта если есть оригинальные тексты

Тексты — память. Карта — зеркало.

```
Тексты хранят полноту. Карта даёт навигацию.
Человек с 500 текстами за 3 года НЕ МОЖЕТ сам:
  - Заметить что "свобода" в 80% случаев рядом со "страхом"
  - Увидеть что определение "любви" изменилось после июня
  - Сравнить своё понимание "ответственности" с чьим-то другим
  - Найти все места где он связывал "работу" со "смыслом"

Карта = автоматически построенный индекс по содержимому текстов.
Каждый элемент содержимого ссылается на оригинальный контекст.
```

### 13.8. FVSC vs LLM

FVSC — не конкурент LLM. Это другой инструмент.

```
LLM:
  - Обрабатывает в момент запроса, ничего не строит
  - Не может ПОКАЗАТЬ структуру — может только рассказать
  - Галлюцинирует, интерполирует, додумывает
  - Интерпретирует (то, что FVSC сознательно не делает)

FVSC:
  - Строит постоянную структуру
  - Каждая связь трассируется до текста
  - Визуализируема, навигируема, сравнима
  - Детерминирована и честна

Вместе (FVSC + Антураж/LLM):
  - LLM получает структурированный контекст из FVSC
  - Это делает LLM точнее чем LLM с сырым текстом
  - FVSC записывает, Антураж интерпретирует
```

### 13.9. Противоречия во времени

Человек в январе: "свобода требует ответственности". В марте: "свобода не требует ответственности". Оба записываются — это реальный конфликт внутри человека.

```
Детекция: тот же контейнер + то же содержимое + тот же глагол,
но ПРОТИВОПОЛОЖНОЕ качество (AFFIRMATIVE vs NEGATIVE).

Запись:
  "свобода" содержит "ответственность":
    элемент 1: (через "требовать", AFFIRMATIVE, timestamp=январь, ×8)
    элемент 2: (через "требовать", NEGATIVE, timestamp=март, ×3)

  conflict = true

Это не ошибка — это информация. Человек изменил позицию,
или находится в состоянии внутреннего конфликта.
Антураж может заметить: "у тебя противоречие в понимании
связи свободы и ответственности — хочешь разобраться?"
```

Конфликт — ценная информация для рефлексии, не баг системы.

> Источник: AGM Theory (Alchourrón, Gärdenfors, Makinson 1985) — теория ревизии убеждений; паранепротиворечивая логика — хранение противоречащих суждений без "взрыва" (Logics of Formal Inconsistency); Temporal Knowledge Graphs (EvoKG) — отслеживание фактов во времени с timestamps.

### 13.10. Родственные формы слова

"Свобода" (сущ.), "свободный" (прил.), "свободно" (нареч.), "освободить" (глаг.) — это РАЗНЫЕ контейнеры. Буквально разные слова с разными леммами и разным употреблением.

```
Каждая форма — свой контейнер:
  "свобода"    — наполняется из контекстов существительного
  "свободный"  — наполняется из контекстов прилагательного
  "освободить" — наполняется как глагол (и связка, и контейнер)

Между ними может возникнуть связь:
  - Через ко-окуренцию (PMI): часто появляются в одних контекстах
  - Через тезаурус: общий корень, словообразовательное гнездо
  - Тип связи: DERIVED_FROM или CO_OCCURS
```

Контейнерами являются все значимые слова (NOUN, VERB, ADJ, ADV). Служебные слова (предлоги, союзы, частицы) — не контейнеры, они структурные элементы, а не носители персонального смысла.

> Источник: MorphoLex — морфологические базы для связи словообразовательных гнёзд; деривационная морфология — словообразование как DAG. Главное: не сливать в один контейнер, т.к. семантический сдвиг при деривации не предсказуем.

### 13.11. Время как свойство связки

Время глагола — свойство ребра, извлекаемое из морфологии.

```
"Я любил свободу"    → tense = PAST
"Я люблю свободу"    → tense = PRESENT
"Я буду любить"      → tense = FUTURE

Элемент содержимого = {
    ...
    время: PAST / PRESENT / FUTURE
    timestamp: 2026-01-15        // когда текст написан
}
```

Время важно для отслеживания эволюции: если человек раньше писал "люблю свободу", а теперь "любил свободу" — его отношение изменилось. Система фиксирует оба факта с их временными метками.

> Источник: морфологический анализ spaCy (Tense feature).

### 13.12. Имплицитные связи

"Свобода. Ответственность. Они неразделимы."

Синтаксически "свобода" и "ответственность" не связаны напрямую — они в разных предложениях. Но связь есть.

```
Механизм 1: Ко-окуренция на уровне абзаца.
  Если два значимых понятия появляются в соседних предложениях
  одного абзаца → CO_OCCURS с пониженным весом.
  Если это повторяется → вес растёт, возможна промоция.

Механизм 2: Разрешение "они/эти/такие".
  "Они неразделимы" — "они" (мн.ч.) указывает на множество.
  Кандидаты: ближайшие существительные из предыдущих предложений.
  Если кандидаты найдены → связать их через предикат "неразделимы".

Механизм 3: PMI подхватит.
  Даже если ни один механизм не сработал при извлечении,
  PMI обнаружит устойчивую ко-окуренцию "свобода" + "ответственность"
  и создаст связь статистически.
```

Принцип остаётся: лучше пропустить и подхватить позже через PMI, чем записать неверную связь.

> Источник: GRACR (2023) — document-level relation extraction через графы; Implicit Discourse Relation Recognition (ACM Computing Surveys) — распознавание связей без явных коннекторов. SOTA требует больших корпусов; наш гибрид (ко-окуренция + PMI + Антураж) — прагматичная альтернатива для малых персональных корпусов.

### 13.13. Пассивный залог

Пассив меняет синтаксические роли, но не семантические. Система должна восстанавливать семантического агента.

```
"Ответственность требуется свободой"
  Синтаксический субъект: "ответственность" (nsubj:pass)
  Семантический агент: "свобода" (obl:agent или instrumental)

  → Детекция: dep_ == "nsubj:pass" → перевернуть направление
  → Записать: "свобода" содержит "ответственность" (как в активном залоге)
```

> Источник: PropBank (Palmer et al. 2005) — ARG0/ARG1 сохраняют роли при смене залога; AMR (Banarescu 2013) — нормализация пассива к активу.

### 13.14. Глагол как контейнер — что внутри

Глагол как **связка** → создаёт ребро (элемент содержимого между понятиями).
Глагол как **контейнер** → накапливает контексты (предложения где использован).

```
"Свобода требует ответственности"
"Жизнь требует мужества"
"Он требует справедливости"

Контейнер "требовать" накапливает эти суждения в ρ(требовать).
Собственные вектора ρ обнаруживают грани:
  грань v₁: абстрактные требования (свобода, жизнь)
  грань v₂: личные требования (он, я)

Содержимое "требовать" формируется из контекстов,
а не из субъектов/объектов (это было бы дублирование рёбер).
```

> Источник: FrameNet (Fillmore 1982) — глагол эвокатор фрейма с собственной структурой; Pustejovsky (1995) — qualia structure: четыре измерения значения слова (formal, constitutive, telic, agentive); VerbNet — суб-событийная декомпозиция глаголов.

### 13.15. Одна пара, разные глаголы

```
"Свобода требует ответственности"     → глагол "требовать"
"Свобода нуждается в ответственности" → глагол "нуждаться"
```

Два РАЗНЫХ элемента содержимого. Глаголы разные — нюанс разный: "требовать" = внешнее обязательство, "нуждаться" = внутренняя потребность.

```
"свобода" содержит "ответственность":
  элемент 1: (через "требовать", ×8)
  элемент 2: (через "нуждаться", ×3)
```

Система сохраняет различие. Для группировки при анализе можно использовать категории (оба → REQUIRES), но при записи — точный глагол.

> Источник: Apresjan — систематическая лексикография, различение близких синонимов по семантическим, оценочным и коннотативным параметрам; тропонимия в WordNet — "X is to Y in some manner".

### 13.16. Составные глаголы

"Привести к" ≠ "привести". Предлог меняет смысл.

```
Глагол + управляемый предлог = один юнит:
  "привести к"    → "привести_к"
  "зависеть от"   → "зависеть_от"
  "состоять из"   → "состоять_из"
  "привести"      → "привести" (другой смысл)

Детекция: spaCy маркирует предлоги как `prep` dependency
при глаголе. Если предлог — прямой зависимый → объединить.
```

Составной глагол — тоже контейнер со своими контекстами.

> Источник: Baldwin & Kim (2010) — Multi-Word Expressions: лексемы, демонстрирующие семантическую идиоматичность; Constant et al. (2017) — обзор MWE в NLP.

### 13.17. Обоснование числовых параметров

Все числовые значения — начальные оценки, подлежащие калибровке на данных.

```
Интенсивность (0.3 / 0.6 / 0.9):
  Начальные значения из лексикографических ресурсов.
  Могут быть уточнены через:
  - RuSentiLex (Loukachevitch 2017): 16000+ слов с оценками
  - NRC-VAD Lexicon (Mohammad 2018): valence-arousal-dominance
  - Корпусный метод: паттерны "X, но не Y" дают шкалу интенсивности
  - Систематическая лексикография Апресяна: ручная градация синонимов

PMI > 0, count ≥ 3:
  Стандартные пороги корпусной лингвистики (Church & Hanks 1990).

Asymmetry > 0.3:
  Эвристика, требует проверки на данных.

Inter-cluster similarity < 0.15 (детекция омонимов):
  Требует калибровки.
```

Все параметры должны быть конфигурируемыми, не захардкоженными.

> Источники: SentiStrength (Thelwall 2010) — градуированный сентимент; NRC-VAD (Mohammad 2018) — Best-Worst Scaling для надёжных оценок; RuSentiLex (Loukachevitch 2017) — русский лексикон; Church & Hanks (1990) — PMI для ко-окуренции.

### 13.18. Прилагательное → существительное

Dependency label `amod` (adjectival modifier): прилагательное модифицирует существительное.

```
"Важная свобода"
  dependency: свобода ←[amod]— важный
  → "свобода" содержит "важный"
    (связка = "amod", AFFIRMATIVE, FACTUAL)

"Очень важная свобода"
  → "свобода" содержит "важный"
    (связка = "amod", интенсивность модифицирована "очень": ×1.3)
  Детекция: advmod("важный", "очень") → модификатор интенсивности

"Свободный человек"
  → "человек" содержит "свободный" (связка = "amod")
```

Связка = `amod` (тип зависимости из текста), а не выдуманный глагол "являться". Стенографический принцип: если глагола нет — не фабриковать.

Прилагательное как контейнер тоже наполняется из своих контекстов: "важный" для одного человека содержит {серьёзный, критичный}, для другого — {ценный, значимый}.

> Источник: Universal Dependencies — `amod`, копула не вставляется при отсутствии в тексте; Palmer, Gildea, Xue (2010) — SRL; PropBank — non-verbal predicates; arXiv 2307.01870 (2023).

### 13.19. Безличные предложения

"Нужно учиться", "Холодно", "Темнеет" — нет грамматического субъекта.

```
Принцип: нет субъекта → нет контейнера → суждение не извлекается.
Контекст сохраняется для кластеризации.

Исключение: если из контекста ясен субъект:
  "Мне холодно" → "я" (дат. падеж) — субъект-переживатель
  → "я" содержит "холодно" (через "быть", AFFIRMATIVE)

  "Нужно учиться" → субъект неопределён → не извлекать.
  Контекст сохраняется. Слово "учиться" активируется как понятие.
```

Согласуется со стенографическим принципом: если субъект не указан самим человеком — система не додумывает.

> Источник: Universal Dependencies — `expl:impers` для нулевых субъектов; UD_Russian-SynTagRus; Moore (UCSD) — Distribution of Russian Impersonals.

### 13.20. Координация (несколько субъектов/объектов)

"Свобода и ответственность требуют мужества" — два субъекта.
"Я ценю свободу, честность, мужество" — три объекта.

```
Принцип: раскрыть координацию — каждый элемент получает
полный набор связей, как если бы был единственным.

"Свобода и ответственность требуют мужества"
  → "свобода" содержит "мужество" (через "требовать")
  → "ответственность" содержит "мужество" (через "требовать")

"Я ценю свободу, честность, мужество"
  → "я" содержит "свобода" (через "ценить")
  → "я" содержит "честность" (через "ценить")
  → "я" содержит "мужество" (через "ценить")

Детекция: dependency label `conj` связывает
координированные элементы. Раскрыть в отдельные суждения.
```

> Источник: Enhanced Universal Dependencies (Schuster et al., Stanford) — раскрытие координации через enhanced deps; SRL as Syntactic Dependency Parsing (EMNLP 2020).

### 13.21. Сравнения

"Свобода важнее безопасности" — ранжирование, не включение.

```
Записать как кортеж:
  {A: "свобода", B: "безопасность", dimension: "важный", comparator: ">"}

На уровне контейнеров:
  "свобода" содержит "безопасность"
    (через "быть важнее", AFFIRMATIVE, FACTUAL)
    comparison = true, direction = ">"

  "безопасность" содержит "свободу"
    (через "быть менее важным чем", AFFIRMATIVE, FACTUAL)
    comparison = true, direction = "<"

Детекция: сравнительная степень прилагательного (Degree=Cmp
в морфологии spaCy) + "чем"/генитив сравнения.
```

> Источник: Ranking entities using comparative relations (SpringerLink); arXiv 2210.13181 — probing comparative correlatives.

### 13.22. Восклицания без предиката

"Какая свобода!", "Вот это да!" — нет связки, нет объекта.

```
Принцип: нет суждения → нет элемента содержимого.

"Какая свобода!"
  → Суждение НЕ извлекается (нет предиката).
  → Понятие "свобода" активируется (контекст сохраняется).
  → Контекст содержит восклицательную конструкцию —
    это может повлиять на кластеризацию (эмоциональная грань).

"Вот это да!"
  → Ничего не извлекается. Нет значимых понятий.
```

Согласуется со стенографическим принципом: если человек не сформулировал суждение — система не может его записать. Восклицание выражает эмоцию, но не определяет понятие.

> Источник: Potts (2008) — exclamatives содержат пропозициональное содержимое, но оно прагматическое, не эксплицитное; Rett (2011) — exclamatives, degrees and speech acts.

### 13.23. Ограничения spaCy для русского языка

spaCy — не идеальный парсер. Ошибки парсинга = ошибки записи. Это признанное ограничение.

```
Точность моделей (ru_core_news_lg):
  POS tagging:        98.9%
  Dependency parsing: UAS 96.2%, LAS 95.1%
  NER:                95.3% F1

Это значит: ~5% зависимостей разобраны неверно.
На 100 суждений — ~5 записаны с ошибкой.
```

**Типичные ошибки:**
- Непроективные деревья (перекрёстные зависимости) — ArcEager не поддерживает
- Свободный порядок слов в русском — парсер может ошибиться с субъектом/объектом
- Безличные конструкции — нет стандартной обработки
- Сложные морфологические формы — ошибки в роде/числе/падеже

**Подход:**
```
1. Принять ~5% ошибок как шум. При достаточном корпусе
   (50+ текстов) правильные паттерны статистически доминируют.

2. Элементы с количеством = 1 — ненадёжны.
   Элементы с количеством ≥ 3 — подтверждены повторением.

3. Пользователь может корректировать через Антураж
   (см. 13.24).

4. Альтернативные парсеры (Stanza, DeepPavlov, Natasha)
   могут использоваться как fallback или для кросс-валидации.
```

> Источник: spaCy Russian models (HuggingFace); Naeval project (github.com/natasha/naeval) — бенчмарки русских NLP-систем.

### 13.24. Коррекция пользователем

Когда пользователь говорит "нет, я имел в виду другое" — карта принимает уточнение.

```
Механизм:

1. Пользователь через Антураж указывает на ошибку:
   "Когда я сказал 'свобода требует', я не имел в виду
    что свобода содержит ответственность"

2. Антураж показывает текущий элемент содержимого.

3. Пользователь уточняет:
   а) Удалить элемент → элемент удаляется, пометка "user_rejected"
   б) Изменить связку → элемент обновляется
   в) Уточнить смысл → добавляется новый элемент
      с пометкой source = "user_confirmed"

4. Подтверждённые пользователем элементы (source = "user_confirmed")
   имеют БОЛЬШИЙ вес чем автоматически извлечённые (source = "user").
   Они не могут быть перезаписаны автоматикой.

Иерархия достоверности:
  user_confirmed > user > citation > auto_inferred
```

Это human-in-the-loop: система извлекает автоматически, человек корректирует при необходимости. Согласуется со стенографическим принципом — уточнение пользователя это тоже его формулировка.

> Источник: Human-in-the-loop knowledge bases; CleanGraph (2024) — interactive KG refinement; Active Learning for NLP (CMU survey).

## XIV. Рекурсивная алгебраическая архитектура

### 14.1. Наблюдение

Система FVSC спроектирована как детерминированная алгебра — без нейросетей, без обучения, без стохастичности. Однако при анализе её структуры обнаруживается глубокий изоморфизм с классом рекурсивных нейронных сетей.

Это не совпадение: и TreeRNN, и FVSC решают одну задачу — **композиция значения из частей**. Но они решают её разными механизмами, и для задачи персональной семантики алгебраический подход строго лучше нейросетевого.

> FVSC — **рекурсивная алгебраическая архитектура**: она разделяет вычислительную структуру Tree-Structured Recursive Neural Network (TreeRNN) и Graph Neural Network (GNN), но заменяет обучаемые матрицы весов детерминированными алгебраическими операциями на матрицах плотности, а backpropagation — накоплением суждений.

### 14.2. Формальная аналогия

| Абстрактная операция | Нейросеть | FVSC |
|---|---|---|
| **Модель** (выученное представление) | Матрицы весов W, обученные через backprop | ρ = Σ wᵢ · \|vᵢ⟩⟨vᵢ\|, накопленная из суждений |
| **Тренировочный пример** | Размеченная пара (x, y) | Суждение S → V → O со свойствами |
| **Обучение** | Градиентный спуск по loss function | Rank-1 обновление: ρ += w · \|v⟩⟨v\| |
| **Выученные фичи** | Активации скрытых нейронов (непрозрачно) | Собственные вектора ρ (интерпретируемо: грани смысла) |
| **Важность фичи** | Величина активации (непрозрачно) | Собственное значение λᵢ (вес грани) |
| **Композиция** (части → целое) | TreeRNN: h = f(W_L · h_left + W_R · h_right + b) | encode(): v = compose(v_subj, T_verb, v_obj) |
| **Message passing** (узел ← соседи) | GNN: h^(k+1) = σ(W · AGG({h_u : u ∈ N(v)})) | ρ_{k+1}(A) = α·ρ_direct + (1-α)·Σ wᵢ·transform(ρ_k(Bᵢ), rᵢ) |
| **Внимание** (что релевантно) | Softmax по key-query dot products | contain(A→B) = Tr(ρ_A · ρ_B) / Tr(ρ_A) |
| **Инференс** | Forward pass через сеть | Trace inner product: Tr(ρ_A · ρ_B) |
| **Сходимость** | Loss plateaus | Неподвижная точка: (1-α)^k → 0 |
| **Регуляризация** | Dropout, L2 | α ∈ (0,1) — прямые компоненты доминируют |
| **Training signal** | Нужен (labels / reward / reconstruction) | **Не нужен** — суждения самодостаточны |
| **Провенанс** | Утерян (распределён по миллионам весов) | **Сохранён** (каждый компонент → конкретное суждение) |
| **Детерминизм** | Нет (random init, dropout, batch order) | **Да** (одни суждения → одна карта) |

### 14.3. Три уровня рекурсии

В системе FVSC действуют три уровня рекурсивной обработки. Каждый изоморфен известной нейросетевой архитектуре.

**Уровень 1 — Внутри предложения: композиция по синтаксическому дереву (аналог TreeRNN)**

Шаг 3 пайплайна обходит дерево зависимостей рекурсивно, от корня к листьям, передавая логический контекст (отрицание, модальность, кванторы). Суждение извлекается с учётом всей цепочки предков в дереве.

Это структурно идентично TreeRNN (Socher et al. 2013), где значение предложения строится снизу вверх из значений потомков. Разница: TreeRNN использует обученную W-матрицу для композиции; FVSC использует детерминированную функцию encode().

> Socher, R. et al. (2013). "Recursive Deep Models for Semantic Compositionality Over a Sentiment Treebank." EMNLP.

**Уровень 2 — Внутри концепта: накопление в ρ (аналог online PCA)**

Каждое суждение о понятии A добавляет rank-1 компонент в ρ(A):

```
ρ(A) += w · |v⟩⟨v|
```

Это математически идентично rank-1 обновлению ковариационной матрицы в online PCA. Матрица плотности ρ — достаточная статистика всех суждений. Eigendecomposition ρ даёт главные компоненты (= грани смысла), как PCA даёт главные компоненты распределения.

Но в отличие от PCA, здесь нет оптимизации: фичи (грани) возникают из накопления, не из минимизации ошибки реконструкции.

> Warmuth, M.K. & Kuzmin, D. (2006). "Online PCA with Matrix Exponentiated Gradient Updates." Rank-1 обновление матрицы плотности как правило обучения.

**Уровень 3 — Между концептами: рекурсивное углубление (аналог GNN message passing)**

```
ρ_{k+1}(A) = α · ρ_direct(A) + (1-α) · Σᵢ wᵢ · transform(ρ_k(Bᵢ), rᵢ)
```

Это структурно идентично message passing в Graph Neural Network (Gilmer et al. 2017):

```
h_v^{(k+1)} = σ(W · AGGREGATE({h_u^{(k)} : u ∈ N(v)}))
```

Маппинг:
- h_v^(k) ↔ ρ_k(A) — состояние узла на итерации k
- N(v) ↔ {Bᵢ : contain(A→Bᵢ) > threshold} — окрестность по containment
- AGGREGATE ↔ взвешенная сумма с весами containment
- σ(W · ...) ↔ α-blend с ρ_direct (без нелинейности, без обучаемого W)
- transform(ρ, r) ↔ edge-conditioned message transformation

Сходимость гарантирована аналогично PageRank: коэффициент (1-α) < 1 обеспечивает затухание.

> Gilmer, J. et al. (2017). "Neural Message Passing for Quantum Chemistry." ICML.
> Kipf, T. & Welling, M. (2017). "Semi-Supervised Classification with Graph Convolutional Networks." ICLR.

### 14.4. Почему алгебраическая форма лучше нейросети для персональной семантики

#### 14.4.1. Отсутствие training signal

Нейросеть требует сигнал ошибки: labels, reward, или reconstruction target. Для персональной семантики такого сигнала не существует.

Кто разметит "правильно ли система записала что свобода значит для тебя"? Сам человек не может — цель системы именно в том, чтобы показать ему то, что он может не осознавать.

- Supervised learning требует (input, output) пар. Output не существует — карта IS the output.
- Self-supervised learning (masked LM) выучит паттерны ЯЗЫКА, не персональные смыслы. Сойдётся к дистрибутивной статистике корпуса, не к индивидуальной семантике.
- Reinforcement learning требует reward function. "Корректность персональной карты" не определена как скалярный сигнал.

FVSC не нуждается в training signal, потому что не оптимизирует. Она накапливает. Каждое суждение самодостаточно — оно одновременно данные и обновление модели. Матрица ρ — не приближение к некоторому таргету; она ЯВЛЯЕТСЯ определением понятия для данного человека.

> Bender, E. & Koller, A. (2020). "Climbing towards NLU: On Meaning, Form, and Understanding in the Age of Data." ACL. — Языковые модели учат форму, не значение.

#### 14.4.2. Провенанс сохраняется структурно

Нейросеть хранит выученные паттерны в распределённых весах. После обучения невозможно ответить: "какое конкретное суждение привело к тому, что сеть считает, что свобода содержит ответственность?" Представление сжато и перепутано по миллионам весов.

FVSC сохраняет провенанс по конструкции. Каждый rank-1 компонент |vᵢ⟩⟨vᵢ| в ρ привязан к конкретному объекту Judgment (субъект, глагол, объект, timestamp, исходный текст). Матрица плотности декомпозируема в составляющие суждения. Это не post-hoc объяснимость — это структурная.

Формально: ρ = Σ wᵢ |vᵢ⟩⟨vᵢ| — **прозрачная модель** в смысле Rudin (2019): структура модели напрямую соответствует данным, из которых она построена, без отдельного метода объяснения.

> Rudin, C. (2019). "Stop Explaining Black Box Machine Learning Models for High Stakes Decisions and Use Interpretable Models Instead." Nature Machine Intelligence.

#### 14.4.3. Детерминизм гарантирован

Нейросети с идентичными данными дают разные модели: random initialization, mini-batch order, dropout masks, GPU non-determinism. Для персональной семантической карты — инструмента рефлексии — недетерминизм фатален. Два запуска на одних данных дают разные карты, и человек не может знать, какая "правильная".

FVSC полностью детерминирована: одни суждения + одни параметры = идентичная ρ, идентичные containment scores, идентичные грани.

#### 14.4.4. Стенографический принцип сохраняется

Стенографический принцип: система записывает ТОЛЬКО то, что следует из формулировок человека. Не интерпретирует.

Нейросеть по определению интерполирует — отображает входы на выученное многообразие, обобщающее за пределы обучающих данных. Это обобщение ЯВЛЯЕТСЯ интерпретацией: сеть "решает", что человек имел в виду, на основании паттернов из других примеров.

В FVSC матрица ρ(A) содержит ровно те суждения, которые человек сделал о понятии A. Рекурсивное углубление добавляет информацию из содержимого A, но эта информация сама трассируется до конкретных суждений. Никакой интерполяции. Никакого обобщения за пределы высказанного.

### 14.5. Накопление vs обучение — как FVSC адаптируется без training

**Определение: Накопление (Accumulation)**

```
ρ_new(A) = ρ_old(A) + w · |v⟩⟨v|
```

Накопление добавляет новый компонент, не модифицируя существующие. После добавления нового суждения каждое предыдущее по-прежнему присутствует в ρ как rank-1 компонент с оригинальным весом. Модель растёт монотонно — не забывает, не перезаписывает, не перевзвешивает. (Recency decay применяется в момент запроса, не хранения.)

**Определение: Обучение (Training)**

```
W_new = W_old − η · ∇L(W_old)
```

Обучение модифицирует ВСЕ веса одновременно на основе глобальной loss function. После обучения отдельные тренировочные примеры не восстановимы из весов. Модель — lossy compression обучающей выборки.

**Стенографический инвариант**: в любой момент времени, для любого понятия A, должно быть возможно перечислить каждое суждение, внёсшее вклад в ρ(A), с оригинальным текстом, timestamp и весом. Накопление сохраняет этот инвариант. Обучение разрушает его.

**Адаптация без обучения** — система адаптируется к конкретному человеку:

1. **Рост разрешения**: больше данных → выше trace(ρ) → больше компонентов → богаче eigendecomposition → больше граней обнаружено. Это не обучение; это увеличение разрешения, как добавление пикселей к изображению.

2. **Вытеснение seed-векторов**: тезаурус даёт начальную структуру. По мере накопления персональных суждений их компоненты доминируют в eigendecomposition (больше по весу и количеству). Seed-вектора не удаляются — они просто оказываются меньшей долей растущей суммы. Аналогично байесовскому prior, который overwhelm'ится данными, но без update rule.

3. **Структурная пропагация**: когда ρ(ответственность) обогащается, ρ(свобода) тоже обогащается на следующем рекурсивном проходе — потому что свобода содержит ответственность. Система становится информированнее по мере накопления данных, но не обучается в оптимизационном смысле.

## XV. Открытые вопросы

1. **"Я" как контейнер — РЕШЕНО.** Реализовано в коде через специальный контейнер [self] с тремя маршрутами:
   - **identity** ("я являюсь свободным", глаголы: являться, быть, стать) → полный вес в [self]
   - **relation** ("я хочу свободу", глаголы: хотеть, любить, бояться) → пониженный вес (×0.5) в [self]
   - **action** ("я делаю X") → НЕ в [self], только активирует объект
   Контейнер [self] не становится свалкой, т.к. action-суждения (большинство) в него не попадают.

2. **transform() в рекурсивном углублении — ОТКРЫТО.** Как преобразовать ρ(B) при вложении в ρ(A) через отношение r? Варианты:
   - Completely positive map (Kraus form из DisCoCirc): Φ_r(ρ) = Σₖ Aₖ·ρ·Aₖ†
   - Проще: R_r · ρ · R_r† где R_r — матрица отношения (строится из ρ(V) глагола)
   - Ещё проще: rotation + scaling (текущая имплементация по сути делает это)

3. **DisCoCat encode v2 — ОТКРЫТО.** v1 использует упрощённое кодирование (сложение + ротация). Цель v2 — полная тензорная свёртка v_j = Σᵢₖ subj_i · T_ijk · obj_k. Вопросы: как строить начальные тензоры T из seed-векторов; как обновлять T из накопленных суждений; какая размерность тензора.

4. **PMI (статистические связи) — ОТКРЫТО.** Не реализован. На малых персональных корпусах PMI шумный. Нужно: Bayesian smoothing, PPMI, минимальный count ≥ 5-8. Нужна спецификация для реальных данных.

5. **Рекурсивное углубление — сходимость — ОТКРЫТО.** При каком α сходимость быстрая и стабильная? Сколько итераций K на практике? Аналоги: PageRank (α=0.85, ~50 итераций), GNN (обычно 2-5 слоёв).

### Оригинальный вклад FVSC (автор: Rein)

Каждый компонент из таблицы выше существует по отдельности. Оригинальность — в архитектуре и принципах, которые их связывают:

1. **Модель контейнера** — значение понятия для человека = то, что он помещает внутрь через свои формулировки. Не Frege (он не предлагал контейнеров), не knowledge graphs (не персональные), не PKM (ручные связи). Конкретная модель "значение = содержимое контейнера" — оригинальна.

2. **Стенографический принцип** — строгое разделение записи и интерпретации как архитектурное решение. Ядро записывает только то, что следует из слов. Интерпретация отдаётся человеку + LLM. Никто не формализовал это для персональной семантической системы.

3. **Глагол как связка И контейнер** — Neo-Davidson говорит "события — сущности", но конкретная идея что глагол наполняется персональным содержимым наряду с ролью связки — оригинальный синтез.

4. **Отпечаток восприятия** — совокупность контейнеров = уникальный отпечаток. Сравнение контейнеров двух людей показывает источник непонимания. Не "кто прав", а "в чём различие определений".

5. **"Лучше потерять связь, чем записать неверную"** — принцип для анафоры и неоднозначностей, следующий из стенографического принципа.

6. **Дискурсивные указатели в контексте персональной карты** — применение DRT к конкретной проблеме: не загрязнять глобальные понятия локальной конкретикой.

7. **Карта отражает восприятие, не словарь** — тезаурус подсказывает, но не диктует. Если для человека омонимы — одно понятие, пусть так. Если полисемия — два разных понятия, пусть разделятся.

8. **Рекурсивный оператор плотности как фрактальный контейнер** — применение density matrix semantics (Coecke, Bankova et al.) к персональной семантической карте с рекурсивной конструкцией: ρ(A) вычисляется из ρ(B₁), ρ(B₂)... содержимого, которые сами определяются из своего содержимого. Это фрактальная, самоподобная структура — один тип объекта (оператор плотности) на каждом уровне вложенности. Содержимое определяет форму контейнера, контейнер может содержать контейнеры, которые содержат его — через разные грани и с разными весами. Ни Coecke, ни Bankova не предлагали рекурсивных density matrices; graph neural networks используют аналогичный message passing, но не на операторах плотности. FVSC — первый проект, соединяющий density matrix semantics с рекурсивной фрактальной конструкцией для персональной карты смыслов.

9. **Рекурсивная алгебраическая архитектура** — формальная демонстрация того, что система структурно изоморфна TreeRNN + GNN, но превосходит нейросетевой подход для персональной семантики по четырём осям: отсутствие необходимости в training signal, структурный провенанс, детерминизм, совместимость со стенографическим принципом. Накопление (accumulation) как альтернатива обучению (training) — с формальным определением стенографического инварианта. Ни одна из существующих систем (TreeRNN, GNN, DisCoCat, AMR) не формализовала это различие для задачи персональной семантики.

### Существующие системы и их ограничения

| Система | Чего не делает (а FVSC делает) |
|---|---|
| AMR (Banarescu 2013) | Не персональный, не накопительный |
| DRT (Kamp 1981) | Не строит карту определений |
| FrameNet (Fillmore 1982) | Прескриптивный, не описывает индивида |
| VerbNet (Levin 1993) | Общеязыковой, не персональный |
| ConceptNet (Speer 2012) | Общий, не персональный |
| Osgood (1957) | 3 оси, ручной ввод |
| Kelly (1955) | Ручное заполнение, не из текста |
| PKM (Obsidian, Roam) | Ручные связи, без автоматического анализа |
| Knowledge Graphs (Wikidata) | Объективные, без субъективности |

## Приложение A. Лексиконы контекстного классификатора

Лексиконы — это знание о грамматическом ПОВЕДЕНИИ слов, не об их значении. Они фиксированы, не зависят от пользователя, и используются контекстным классификатором (шаг 3 пайплайна) для определения референциального статуса NP.

### A.1. Тип предиката

**ILP-прилагательные** (individual-level, постоянные свойства → GENERIC clause):
```
смертный, разумный, свободный, живой, сильный, слабый,
способный, достойный, виновный, ответственный, честный,
умный, глупый, добрый, злой, красивый, прекрасный,
счастливый, несчастный, одинокий, важный, нужный
```

**SLP-глаголы** (stage-level, события → EPISODIC clause):
```
прийти, уйти, упасть, сказать, увидеть, взять, дать,
купить, продать, найти, потерять, позвонить, написать,
отправить, получить, начать, закончить
```

### A.2. Маркеры обобщённости и конкретности

**Маркеры хабитуальности** (→ GENERIC/HABITUAL clause):
```
обычно, часто, всегда, иногда, редко, бывает,
случается, порой, как правило, нередко
```

**Маркеры темпоральной конкретности** (→ EPISODIC clause):
```
вчера, сегодня, завтра, потом, тогда, сейчас,
только что, недавно, утром, вечером
```

### A.3. Детерминеры и классификация NP

**Референциальные детерминеры** (quasi-generic noun + детерминер → REFERENTIAL):
```
этот, тот, один, какой-то, некий, такой, другой,
мой, твой, наш, ваш, свой, его, её, их
```

**Quasi-generic существительные** (требуют контекстной классификации):
```
человек, люди, вещь, штука, место, раз, часть, пара,
тип, чувак, девушка, парень, мужчина, женщина,
ребёнок, друг, знакомый
```

**Абстрактные существительные** (почти всегда CONCEPTUAL):
```
свобода, любовь, ответственность, справедливость,
истина, правда, ложь, счастье, страдание, смысл,
жизнь, смерть, время, пространство, сознание,
мышление, душа, разум, воля, красота, добро, зло,
вера, надежда, мужество, честь, долг, достоинство
```

### A.4. Модальные оболочки

| Глагол | Тип модальности | Вес | Neg-raising |
|---|---|---|---|
| знать | FACTUAL | 1.0 | нет |
| верить | EPISTEMIC | 0.5 | нет |
| думать | EPISTEMIC | 0.5 | **да** |
| считать | EPISTEMIC | 0.6 | **да** |
| полагать | EPISTEMIC | 0.5 | **да** |
| хотеть | DESIDERATIVE | 0.6 | нет |
| мечтать | DESIDERATIVE | 0.5 | нет |
| бояться | EPISTEMIC | 0.4 | нет |
| сомневаться | EPISTEMIC | 0.3 | нет |
| надеяться | DESIDERATIVE | 0.5 | нет |

Neg-raising: отрицание при глаголе-оболочке пробрасывается во внутреннюю клаузу.
"Не думаю, что X" → X получает quality=NEGATIVE.
"Не знаю, что X" → X получает modality=EPISTEMIC, quality не меняется.

### A.8. Маршрутизация [self]

**Глаголы идентичности** (→ [self], полный вес):
```
являться, быть, стать, становиться, считать,
казаться, оказаться, выглядеть, представлять
```

**Глаголы отношения** (→ [self], вес × 0.5):
```
хотеть, любить, ненавидеть, бояться, уметь, понимать,
верить, чувствовать, ощущать, стремиться, нуждаться,
ценить, предпочитать, уважать, презирать, искать,
ждать, надеяться, мечтать
```

**Глаголы действия** (все остальные → НЕ в [self]):
Действия говорят о том что человек делает, но не определяют кто он.

### A.9. Дискурсивные указатели

```
проблема, вопрос, суть, дело, причина, цель,
задача, смысл, итог, вывод, результат, следствие,
условие, основа, главное, важное, интересное
```

Паттерны указательного использования: "{слово} в том, что...", "{слово} заключается в...", "{слово} состоит в...".
При обнаружении: НЕ наполнять контейнер указателя, направить содержимое в реальные понятия.

> Все лексиконы — начальные оценки, подлежащие расширению на данных. Принцип: лучше пропустить и подхватить позже, чем записать неверно.
